# Mirror Abyss 1.2.0-rc.34 多事件线记忆漏斗修复报告

## 修复目标

rc.34 解决 rc.33 之后仍阻挡实机同步验证的表格与总结结构问题：状态请求过重、一次正文可能包含多条无关事件、同一事件会影响多个对象、总结产生额外世界书触发条目，以及当前/近期/历史缺少统一沉降层级。

本轮没有把所有剧情混进一个条目，也没有把一个对象拆成多个触发条目。最终规则是：

> 生成可以批量，事件不能混线；一次事件可以影响多个条目，每个对象或事件仍只有一个完整世界书条目和一种召回方式。

## 一、条目内记忆漏斗

八类默认表统一增加并明确以下层级：

- `baseContent`：对象定义，变化极少；
- `currentFacts`：当前客观成立的现行事实；
- `currentStates`：当前正在发生或阶段性的状态，细节最多；
- `recentHistory`：小总结维护的近期经历，保留主要过程、直接因果和未决影响；
- `solidifiedHistory`：大总结维护的历史事实，只保留最终结果、核心因果和长期影响。

状态提取只允许更新对象定义、现行事实和当前状态，不得直接写近期或历史层。

## 二、一次生成多条独立事件线

状态协议升级为 `MirrorAbyssMultiEventPatchV34`：

- 同一轮正文允许返回多个不同 `event_id`；
- `facts.related_entities` 明确列出直接受影响对象；
- 同一人物、地区或全局对象可以同时接收多条事件线影响；
- 插件本地派生并归并 `factIds / eventId / eventIds / recall`；
- 模型只输出本轮事实变化和对象行级补丁，不重写完整表格。

无关事件只共享一次 API 调用，不共享事实、因果、参与者、总结或事件条目。

## 三、同一事件传播多个条目

例如“屠龙少年杀死最后一条龙”可以同时改变：

- 屠龙少年角色条目；
- 最后一条龙事件/角色条目；
- 北境龙谷地区条目；
- 龙族对象条目；
- 世界格局全局条目。

这些条目可以共享来源 `event_id / fact_id`，但仍是不同信息点。rc.34 删除了旧的“共享 fact/event ID 即视为重复”的跨对象去重方式，改为仅按稳定条目 key 和完全相同正文去重。

## 四、小总结与大总结

小总结和大总结没有删除，而是改为条目内沉降器：

- 每次尽量批量处理所有到期事件线，当前安全上限为 8 条；
- 模型必须对每个请求的 `event_id` 返回且只返回一项；
- 小总结把结果写入同一事件和真实受影响对象的 `recentHistory`；
- 大总结移除该事件已消费的近期版本，并把长期结果写入 `solidifiedHistory`；
- 不同事件的小总结数量不能合并凑大总结阈值；
- 一个事件结束不代表同批其他事件也结束；
- 失败时回滚本批快照和总结凭证，不重跑已经完成的 state。

总结 artifact 继续作为消费、版本和历史恢复凭证，但不再发布为世界书信息点。

## 五、事件条目不再被删除

旧沉降逻辑可能在事件结束后删除事件行。rc.34 只允许清理已经退出的旧时空切片；事件和地区条目继续保留，并在同一条目中承载当前、近期和历史。

因此两个无关事件会保持两个事件条目，不会被压进一个公共大总结。

## 六、单条目单召回方式

正式世界书只发布对象和事件表格条目：

- 玩家焦点角色，以及既有连续性规则要求的条目：`constant`；
- 人物、地区、物品、组织等对象：名称和稳定别名 `trigger`；
- 启用语义时，事件条目整体：`vector`；
- 关闭语义时，事件条目回退为 `trigger`。

`small:event:* / large:event:*` 仅作为旧数据清理兼容键存在，不再生成新的正式条目。

## 七、状态请求减重

rc.34 的默认限制：

- system prompt 约 3,083 字符；
- JSON Schema 约 8,098 字符；
- 输出上限 2,000 tokens；
- 旧行索引最多 120；
- 相关旧行全文最多 16、每表最多 6；
- 内部事实最多 18；
- 本地派生元数据完全退出 Schema；
- 当前事实和状态上下文最多，近期经历次之，历史事实最少。

这降低了请求负担，但本地测试无法证明用户实际网关的约 60 秒 504 已彻底消失。

## 八、回归修复

同时修复：

- 完全锁定条目在总结迁移时可能被删除旧近期文本；
- 总结旧文本可能从无关事件条目中被同文误删；
- 一批事实触发多个小总结时旧测试只允许一个总结；
- 世界书测试仍假设存在独立小/大总结条目；
- 旧全局大总结逻辑会把无关事件混线。

## 十、实机尚待确认

必须用 rc.34 实机验证：

- 两条无关事件是否由一次 state 请求完整返回；
- state 请求是否不再在表格阶段 504；
- 多对象补丁是否正确写入表格并同步世界书；
- 小总结/大总结批量请求是否按事件线分区；
- 世界书是否只保留对象和事件条目，没有新增总结条目；
- rc.33 的恢复后重复任务竞态仍未复发。
