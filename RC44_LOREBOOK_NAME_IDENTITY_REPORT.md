# rc.44 世界书名称唯一性与 UID 锁定修复报告

版本：`1.2.0-rc.44`  
流水线：`ma-pipeline-46`

## 实机问题

同一聊天的世界书中出现两条名称与正文相同的镜渊托管条目。“按对象清理并重新发布”执行后重复项仍保留。

## 根因

旧发布器主要依赖 `managedKey` 与 `logicalKey`：

- 同一对象因旧稳定 ID、迁移或缓存回写产生不同 `managedKey`；
- 多条候选共享或缺失 `logicalKey` 时，被判定为歧义并保护；
- 维护按钮只清理旧格式 key、旧总结 logicalKey 和历史备注范围；
- 现代 key 的同名重复条目不属于上述清理范围。

因此按钮可以完成保存，但并未真正删除这些重复项。

## 修复规则

当前聊天中的镜渊托管条目正式采用以下身份顺序：

1. 当前 `chatKey + managedKey` 命中：视为已经锁定的稳定 UID；
2. 当前 `chatKey + 规范化条目名称`：同名多条时保留一个 UID，删除其余；
3. 当前 `chatKey + logicalKey` 新旧两侧唯一；
4. 无候选时创建新条目。

名称来自 SillyTavern 世界书 `comment`，规范化内容包括：

- Unicode NFKC；
- 移除镜渊 `[MA11]` 前缀；
- `｜` 与 `|` 统一；
- 合并多余空白；
- 拉丁字母大小写统一。

完整名称参与身份，例如“角色｜沉秋”与“物品｜沉秋”不会合并。

## UID 选择

- 已有当前 `managedKey` 的条目优先保留；
- 没有稳定 key 命中时，确定性保留最小 UID；
- 保留条目写入当前 key、logicalKey 和正文；
- 后续正文变化继续覆盖该 UID，不创建新条目。

## 安全边界

- 只删除当前 `chatKey` 的镜渊托管条目；
- 其他聊天条目不处理；
- 无镜渊托管元数据的人工条目不处理；
- 不同完整条目名称不处理；
- desired 侧本身出现同名对象时，禁用名称归并，继续使用 key/logicalKey，避免误合并异常上游数据；
- 正文不再作为对象身份，正文不同也不妨碍同名重复项清理。

## 按钮修复

“按对象清理并重新发布”与普通同步共用名称唯一规则：

- 预览会把当前聊天的同名现代托管副本计入可清理数量；
- 执行时优先保留当前稳定 key 的 UID；
- 删除其余同名托管副本；
- 用当前对象数据覆盖保留条目并保存；
- 保存后回读和编辑器重载阶段继续执行同一归并，防止缓存回流再次产生副本。

## 验证

新增 `tests/rc44-lorebook-managed-duplicates.ts`，验证：

- 名称分隔符、空白和大小写规范化；
- 同名、正文不同、logicalKey 不同仍归并；
- 无稳定 key 时保留最小 UID；
- 后续正文变化复用保留 UID；
- 维护按钮优先保留正确 managedKey UID；
- 其他聊天、人工条目和不同完整名称不被删除；
- desired 侧名称不唯一时不执行名称误合并；
- 旧 logicalKey 歧义保护在名称不匹配时继续生效。

完整 `npm run verify`、20 步模拟、TypeScript 检查、浏览器构建与语法检查均通过。
