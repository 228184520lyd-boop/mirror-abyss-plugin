# Mirror Abyss / 镜渊

版本：`1.2.0-rc.20`　流水线：`ma-pipeline-23`

镜渊是 SillyTavern 前端扩展。它在角色正文生成完成后执行规则审核、必要的定向修正、内部事实提取、动态状态视图、按事件线的小总结、增量大总结和当前聊天世界书发布。插件不生成主叙事正文，不保存供应商密钥，也不替代 SillyTavern 的连接系统。

## 自动处理顺序

```text
角色正文
→ 审核
→ 必要时定向修正并复审
→ 内部事实层 + 启用的动态表格视图
→ 按 event_id 生成小总结
→ 用上一版大总结 + 未固化小总结生成新大总结
→ constant / trigger / vector 世界书发布
```

事实与状态先提交；总结或世界书失败不会回滚核心结果。消息下方以连续阶段卡展示进度，自动流程为主，单步操作默认收在排错工具中。

## 动态表格

默认注册十张表：焦点、时空、状态、物品、技能、显著关系、活跃事件、相关区域、全局变化、相关基础设定。

在“表格管理”中可以新增、删除、重命名、修改用途与字段、上下排序、启用/停用，以及恢复默认十表。表格数量不设固定上限。下一次状态提取的提示词和 JSON Schema 只包含当前启用表格；模型返回未注册或已停用表格会被拒绝。

删除或停用表格只影响可见视图，不删除聊天级内部事实、`event_id`、小总结、大总结或历史重建依据。

## 内部事实层

聊天级事实层独立于表格，保存稳定 `fact_id`、`event_id`、来源消息、已发生事实、未解决事项、状态、时间范围、关联对象，以及小总结消费和大总结固化标记。

状态提取只保留当前焦点周围、对后续生成有作用的已显影事实。纯围观、喝彩、议论、听见或同场不会自动建立人物、关系、事件或附属物。两人对战只保留交战双方、对战事实，以及确实改变战局或结果的第三方。

## 分层总结

小总结按单一 `event_id` 汇总已经发生的事件线，保留起因、关键经过、变化、当前结果、未解决事项和后续影响。进行中的事件只总结已确定部分，不提前写死结果；只有最新事件状态明确结算且没有未决事项时，代码才允许相关可见视图安全退出。

大总结的输入仅为上一版大总结和尚未固化的小总结。成功写入并回读后，对应小总结和事实才标记为已固化；同一小总结不会重复消费。

## 世界书召回

镜渊托管条目只使用三种形式：

- `constant`：极少量当前焦点、时空、必要规则、不可缺失状态和直接相关全局变化。
- `trigger`：人物、别名、物品、技能、区域、事件、组织、制度等明确条件；支持 `any / all / exclude`。
- `vector`：未固化小总结、最新累计大总结及难以依靠关键词恢复的长期因果。

发布流程为常驻 → 明确触发 → 向量候选 → 按 `fact_id / event_id` 去重 → 按总容量裁剪。数值 `order` 只用于世界书编辑器显示顺序，不参与记忆取舍。

SillyTavern 的 World Info 条目仅提供 `vectorized` 标记；实际向量分数门槛和 Max Entries 由 Vector Storage 全局设置控制。镜渊保存期望门槛等托管元数据，并可限制自己发布的向量候选数量，但不会擅自修改全局向量设置。

## 历史编辑与重建

- 正文与审核规则未变化时复用审核结果。
- 从第一个受影响状态开始重建，不执行无差别全链 force 重跑。
- 只移除来源受影响的事实、事件线总结和后续累计总结。
- 小总结只重新消费符合条件的受影响 `event_id`；大总结只消费未固化小总结。
- 世界书重新发布不调用模型。
- 编辑、删除、聊天切换、禁用、重启和重置都会使旧任务失去提交资格。

## rc.19 数据迁移

首次读取旧聊天时自动执行：

- `characters → 状态`
- `relationships → 显著关系`
- `events → 活跃事件`
- `regions → 相关区域`
- `foundations → 相关基础设定`
- 新增空的“全局变化”视图
- 旧人工行和锁定行保留
- 旧 artifact 事实迁入内部事实层
- 旧小总结/大总结消费关系继承，避免重复固化
- 只操作带镜渊托管标记且 `chatKey` 完全一致的世界书条目，人工条目不受影响
