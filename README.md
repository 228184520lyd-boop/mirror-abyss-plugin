# Mirror Abyss / 镜渊

版本：`1.2.0-rc.42`　流水线：`ma-pipeline-44`

镜渊是 SillyTavern 前端扩展。它在角色正文生成完成后执行规则审核、必要的定向修正、内部事实提取、对象化状态视图、按事件线的小总结、增量大总结和当前聊天世界书发布。插件不生成主叙事正文，不保存供应商密钥，也不替代 SillyTavern 的连接系统。

## 自动处理顺序

```text
角色正文
→ 审核
→ 必要时定向修正并复审
→ 内部事实层 + 启用的对象视图
→ 按 event_id 生成小总结
→ 用上一版大总结 + 未固化小总结生成新大总结
→ constant / trigger / vector 世界书发布
```

事实与状态先提交；总结或世界书失败不会回滚核心结果。自动流程为主，单步操作默认收在排错工具中。



## rc.42 业务链路收敛

本版只删除可复现的重复链路，不合并审核/状态模型职责，不删除派生独立失败和聊天隔离保护：

- 合规正文聊天保存由 6 次收敛为 3 次；
- 修正正文聊天保存由 8 次收敛为 5 次；
- 历史恢复每条正文只在完成后推进一次进度；
- 最新正文恢复将核心状态与解除历史暂停合并为一次 metadata 提交；
- 核心提交结果直接传给派生资格判断，世界书同步复用已读取 ChatState；
- 旧派生任务按 `automatic + kind` 显式分类取消，不再依赖 key 字符串。

模型调用顺序保持：合规正文 `audit → state`，修正正文 `audit → revision → audit → state`。完整证据见 `RC42_CHAIN_SHORTENING_REPORT.md`。

## rc.41 基础设施契约差分修复

本版不改表格、记忆漏斗、审核规则、总结规则或世界书布局，只修复通过失败测试确认的基础设施 Bug：

- 请求执行函数同步抛错时，lane 会永久卡死；
- structured 与 audit fallback 依赖修复模型后仍错误缓存为以后跳过 Schema；
- audit 遇到 5xx、超时、鉴权、限流或取消时仍追加 plain 请求；
- 阶段重新排队或直接阻断时继承旧时间戳、旧错误或错误 attempts；
- 禁用和运行时重置未给 active 任务留下明确取消元数据；
- 已请求取消的任务与迟到 502/504 竞态时会被误记为 failed；
- 插件重启会清空仍在执行的世界书物理写入锁；
- ChatState 读取暴露 metadata 活对象，未提交修改可泄漏；
- ChatState 保存或清空失败时，内存 metadata 保留未落盘的假状态。

修复证据与边界见 `RC41_FOUNDATION_CONTRACT_REPORT.md`。

## rc.40 Schema 拒绝缓存提交时机修复

rc.39 实机历史恢复出现：状态请求直接走 `plain`，模型输出在 4096 tokens 前被截断，后续重试仍继续绕过已经实机成功的 `MirrorAbyssStateOperationsV37`。根因是具体 Schema 的拒绝缓存写入过早：只要无 Schema fallback 的网络请求返回成功，就立即缓存，尚未确认返回 JSON 是否完整、能否通过本地业务 Schema。

rc.40 只修复这条错误缓存链：

- 具体 Schema 拒绝只有在 fallback 完整解析并通过本地业务校验后才提交；
- fallback 被截断、JSON 无法解析或业务 Schema 不合格时，不写入拒绝缓存；
- 已被缓存绕过的 plain 输出一旦失败，立即清除该具体 Schema 缓存，下一次重试重新走严格 Schema；
- 具体 Schema 拒绝不再持久化到 `sessionStorage`，仅在当前页面内存中复用；
- 升级时清除 rc.35–rc.39 遗留的持久化具体 Schema 拒绝记录；
- 连接级“明确不支持结构化输出”缓存保持不变；
- 审核和连接测试使用相同的“验证成功后再缓存”规则。

状态 Schema、状态最大输出、表格协议、审核规则、历史恢复流程和世界书逻辑均未修改。详细证据见 `RC40_SCHEMA_CACHE_VALIDATION_REPORT.md`。

## rc.39 请求调度与历史状态边界

rc.39 保留 rc.37 已实机通过的严格状态 Schema 和 rc.38 固定台账修复，只调整请求调度、错误回退和历史状态边界：

- 同一物理连接拆分为串行 business 通道与串行 diagnostic 通道；连接测试不再等待长状态请求，也不会插入状态 Schema 与兼容回退之间；
- HTTP 5xx、超时、鉴权、限流和取消不再触发无 Schema 第二次业务请求；只有明确 Schema 兼容错误才允许一次回退；
- 活动请求按 task/requestClass 登记，新正文只取消自动总结业务请求，历史失效只取消业务请求，诊断不会被连带终止；
- 历史恢复进度不再改写上一次真实世界书同步结果；同步卡片在恢复期间显示暂停；
- 旧版被历史控制流污染的 lastSync 状态自动迁移，暂停旧世界书失败会持久记录；
- 请求诊断新增 connectionLane、requestClass、requestOrigin。

详细证据见 `RC39_REQUEST_SCHEDULING_REPORT.md`。

## rc.38 固定台账收口

rc.38 不改变已经在实机上直接成功的 `MirrorAbyssStateOperationsV37`。本版回到固定 48 项问题台账，集中修复生命周期、任务终态、自动总结阻塞、中断状态残留、响应信封、错误分类和旧设置迁移：

- 重复激活不再累积启动定时器或旧事件监听器；
- 页面刷新、重启或切回聊天时，遗留 `queued/running` 阶段对账为 cancelled，历史恢复对账为 failed 并保留重试入口；
- 无业务产出的自动大总结不再创建 success 空任务；
- 新正文可安全取消同聊天运行中的自动小/大总结，让 state 优先执行，但仍保持单 active、无并发写入；
- 聊天切换、提交守卫拒绝和插件关闭统一记为 cancelled/skipped/blocked，不再以 null 返回伪装 success；
- 扩展模型响应信封识别，独立处理 No message generated、refusal、incomplete 和工具调用参数；
- 诊断新增 `errorKind/httpStatus/cancelReason`；旧 `auditFailAction: delete` 自动迁移为 hide；
- TypeScript noEmit 检查正式加入打包回归。

完整 48 项状态和统一实机步骤见 `RC38_MASTER_LEDGER_REPORT.md`。

## rc.37 严格扁平状态 Schema

rc.36 实机再次证明：仅把状态 Schema 从约 10 KB 缩到约 2.6 KB 仍会在约 2 秒内返回 400；同连接的审核 Schema 持续成功。因而“单纯过大、过深”不是完整根因。最显著的剩余差异是：审核使用 `strict: true` 且所有对象属性全部列入 `required`，rc.36 状态 Schema 使用 `strict: false` 并依赖大量可选动态属性。

- 状态供应商 Schema 改为 `MirrorAbyssStateOperationsV37`，使用 `strict: true`；
- Schema 中每一级 object 的全部 properties 均进入 required，并设置 `additionalProperties: false`；
- facts 的时间范围改为扁平 `time_start / time_end / time_label`，入库前本地恢复为原 `time_range`；
- 动态表格字段改为 `fields: [{ key, values }]`，字段是否存在、类型是否正确、能否用于目标表，全部由镜渊按当前注册表本地验证；
- lifecycle 使用固定完整对象；无变化时输出空值，插件不会用空生命周期覆盖旧值；
- 保留 rc.36 直接字段输出和旧 snapshot 输出的兼容读取；
- 诊断导出现在保留 `jsonSchemaName / jsonSchemaBytes`，便于下一轮实机精确确认实际发送版本；
- 完整八表 Schema、对象定义保护、总结字段禁写和世界书事务边界均未改变。

默认状态传输 Schema 约 2,304 字节、35 个属性。详细证据见 `RC37_STRICT_SCHEMA_REPORT.md`。

## rc.36 状态 Schema 传输层瘦身

rc.35 实机证明：同一连接的审核 Schema 成功，而完整状态 Schema 在约 2 秒内返回 400，普通 JSON 回退随后成功。rc.36 据此先验证“复杂度过高”假设并缩减传输 Schema；后续 rc.36 实机显示缩减后仍返回 400，因此该假设只解释了风险，不足以单独解释当前接口拒绝。

- 供应商侧状态输出改为单一 `operations` 列表，当前只允许显式 `upsert`；
- 无变化对象不输出 operation，未返回行保留，不提供隐式清表或模型侧删除；
- 完整八表动态 Schema 留在镜渊本地，operations 归一化后再进行完整业务校验；
- 默认传输 Schema 从约 10,114 字节、147 个属性缩减为约 2,640 字节、42 个属性；
- 请求诊断新增 `jsonSchemaName / jsonSchemaBytes`；
- 历史恢复运行时，普通手动处理不再排入恢复后的陈旧 state 任务。

详细证据见 `RC36_STATE_SCHEMA_REPORT.md`。

## rc.34 多事件线记忆漏斗与表格减重

rc.34 将状态、总结和世界书统一到“一名一条、一条一触发”的模型：

- 一轮正文可在一次状态请求中识别多条互不关联的事件线，并为人物、事件、地区、物品、全局等多个受影响条目返回行级补丁；
- 同一对象可同时接收多条事件线影响，插件按 `event_id / fact_id / related_entities` 本地归并，不允许后一个事件覆盖前一个事件；
- 八类对象表新增 `currentFacts`（现行事实）与 `recentHistory`（近期经历），形成“当前最详细、近期次之、历史最精简”的条目内记忆漏斗；
- 状态提取只维护对象定义、现行事实和当前状态；`recentHistory / solidifiedHistory` 只由小总结和大总结维护；
- 小总结和大总结尽量一次请求批量处理所有到期事件线，但输出必须按 `event_id` 分区，两个无关事件绝不混合；
- 小总结写回同一事件及其真实受影响对象的近期经历，大总结将已成熟近期内容移入历史事实；总结 artifact 仅保留为内部凭证，不再生成额外世界书条目；
- 已结束事件条目继续保留，不再因沉降从事件表删除；一个事件线始终拥有自己的事件条目和触发配置；
- 同一事件传播到人物、地区、龙族和全局时，各独立条目全部保留，世界书不再以共享 `fact_id / event_id` 误判为重复；
- 召回保持单条目单模式：焦点或连续性条目为 constant，人物/地区/物品等对象按名称和别名 trigger，启用语义时事件条目整体 vector；
- 状态上下文上限收紧为 120 条索引、16 条相关全文、每表最多 6 条、18 条内部事实；状态输出上限为 2000 tokens；本地派生元数据退出模型 Schema。

默认状态 system prompt 约 3,083 字符，Schema 约 8,098 字符。该修改降低了请求输入输出，但是否彻底避开实际站点约 60 秒的上游 504，仍需 rc.34 实机诊断确认。

详细证据见 `RC34_MULTI_EVENT_MEMORY_REPORT.md`。

## rc.33 历史恢复自动任务竞态

rc.33 不改变模型提示、状态表内容、总结规则或世界书条目结构，只修复 rc.32 实机发现的恢复后重复处理：

- 历史恢复存在时，`MESSAGE_RECEIVED`、编辑和 swipe 等自动调度不再创建普通处理任务；
- 恢复开始与提交前会清理当前聊天尚未触发的自动定时器；
- 恢复正式提交前，取消恢复期间由宿主保存事件排入的相同正文自动任务；
- 自动任务真正启动时再次校验正文指纹、消息键、状态成功记录和 `processedMessageKeys`；相同输入已经提交则记为 `skipped`，不再调用模型；
- 新增 `TaskSkippedError` 与任务终态 `skipped`，区别于历史锁的 `blocked` 和生命周期取消的 `cancelled`；
- 诊断新增 `triggerSource`、`messageKey`、`messageFingerprint`、入队历史修订、入队恢复阶段、`automatic` 和 `skipReason`；
- 流水线版本升级为 `ma-pipeline-35`，避免旧页面运行期任务键与新逻辑混用。

详细证据见 `RC33_RECOVERY_RACE_REPORT.md`。

## rc.32 历史事务闭环

rc.32 不改变审核、状态内容、总结规则或世界书条目结构，只修入口权限与提交终态：

- 新增 `blocked` 任务语义；历史锁阻止的操作不再被标成 `success`；
- 历史失效期间，普通整理、state 重试、summary 和 sync 均不能修改正式状态或世界书；
- `force` 只表示重新执行当前阶段，不再拥有绕过历史锁的能力；
- 只有历史重建函数持有显式恢复权限，普通入口无法伪造恢复提交；
- 历史核心状态重建后不提前解除锁，世界书真正保存并回读确认后才解除；
- 总结或世界书恢复失败时保留历史锁和恢复阶段，禁止半完成状态被视为成功；
- 新增两组生产级回归，覆盖“普通任务越锁”和“保存完成前不得解锁”。

详细证据见 `RC32_HISTORY_TRANSACTION_REPORT.md`。

## rc.31 行级状态补丁与请求压缩

rc.31 针对 rc.30 实机仍触发 60 秒上游 504 的状态链继续收敛：

- 状态模型不再返回“变化整张表”，改为只返回新增或变化行；插件按稳定 ID 本地 upsert，未返回行继续保留；
- 显式空数组仍表示确认清空整表，避免把“未返回”误判为“空表”；
- 上一快照改为“稳定 ID 索引 + 相关行全文”，不再把所有旧行正文全部塞入模型；
- 默认状态系统提示由约 10,435 字符压缩到约 2,528 字符；
- 默认 JSON Schema 由约 15,300 字符压缩到约 10,188 字符；
- 状态最大返回额度由 4096 收紧到 2400，降低长思考与完整输出触碰 60 秒网关的概率；
- 同一连接的 Schema 不兼容结论跨 audit/state 等任务共享，并写入 sessionStorage，页面刷新后仍可跳过已知失败的 Schema 请求；
- 诊断新增 requestPurpose、systemPromptChars、promptChars、responseTokens 与 hasJsonSchema，后续能直接判断请求体和回退阶段。

该修改不重写审核、总结、世界书或存储结构。旧模型返回完整表时也会按行覆盖兼容处理。

## rc.30 实机恢复链修复

- JSON Schema 请求明确返回 `Bad Request/400` 时，立即记住当前连接不支持该结构化方式；即使无 Schema 回退随后 504，下一次也不会重复发送已知失败的 Schema 请求。
- 状态 `snapshot` 改为增量表格视图：只返回本轮变化的整张表；未返回的启用表格沿用上一份，只有显式 `[]` 才代表清空。
- JSON 在对象闭合前被截断时直接报告“输出被截断”，不再调用模型修复缺失内容。小型标点、围栏等格式问题仍保留有限本地解析与一次可选修复。
- 状态提交前强制验证 `turnSummary`、`facts`、`snapshot` 及返回表格类型；验证失败不会覆盖旧快照。
- 历史重建新增核心状态、派生恢复、世界书发布、失败/部分完成状态，诊断和 UI 会显示真实失败位置。
- 模型失败后先通知 UI，再等待 SillyTavern 保存失败状态，降低移动端长时间停留在“处理中”的错觉。

## rc.29 可编辑语义表头

默认表头现在只是初始模板。玩家可以在“表格管理”中修改 `title / content / status / keywords` 的显示名称和传输说明：

- 新表头进入下一次状态提取提示词和 JSON Schema 描述；
- 主表列头、对象编辑器四个主字段标签与世界书正文使用新表头；
- JSON、快照和世界书身份继续使用稳定 `table.key / field.key / row.id`；
- 改表头不会迁移、重命名或清空旧对象值；
- 默认表头没有变化时，世界书正文保持 rc.28 格式；
- 状态请求进行中若表格定义变化，旧 Schema 结果会被提交守卫拒绝；
- 旧请求被拒绝后表格阶段恢复为可重试状态，不会永久停在“处理中”；
- 已有自定义字段不能在一次编辑中伪装成改键，字段类型也不能静默改变。

表格定义仍是全局扩展设置，适用于所有聊天；本版没有把注册表迁入聊天状态，也没有新增迁移位。

## rc.28 核心稳定与最小减法

rc.28 不增加对象字段、叙事能力或新迁移位，只收窄五条核心热路径并补生产链回归：

- 焦点不再根据历史 artifact、旧 ID 或名称推断；当前快照中不是现有角色的焦点直接清除。
- 状态缓存指纹由完整规范化请求生成；旧 artifact 没有指纹时视为未命中。
- 标题、`eventId`、`factId` 回退只在新旧两侧均唯一时使用；人工基础和完全锁定不在同名歧义间转移。
- 事实、快照和世界书发布共用一个纯旁观/因果介入谓词。
- create、sync、pause、clear 按最终物理世界书名称共享 mutation gate；`chatKey` 仍只负责归属和删除保护。
- 普通同步只按当前聊天唯一 `managedKey`、唯一 `logicalKey` 匹配，否则创建；备注、精确签名和旧摘要键只在玩家主动“清理并重新发布”维护流程中检查。
- 普通同步无变化时不保存、不刷新；实际保存后回读一次，只有真实打开世界书编辑器后才做刷新后确认。
- `CURRENT_SCHEMA_VERSION = 2` 且 V23–V30 必要兼容位均完成时，读取走轻量快路径，不扫描、深拷贝或比较全部历史 artifact。
- 当前数据写入不再反复执行旧总结消费关系迁移，事实语义更新后可以正确生成下一版小总结。

没有新增 V31/V32。V23–V30 转换器、TaskQueue、RequestLaneScheduler、提交守卫、聊天隔离、人工基础保护、完全锁定和 base/current 分层全部保留。


## rc.25 注释与全面修改审计

rc.25 不重写既有架构，补充 39 个 TypeScript 模块的职责/维护边界，并对 rc.23—rc.24 的关键链路逐项复核。注释集中在对象迁移、基础保护、摘要版本链、稳定对象键、共享世界书保护、刷新后回读归并和唯一焦点常驻，不做逐行翻译式注释。

全面审计确认两处 rc.24 没有完全落到代码：一是区域分配只删除与当前时空路径重叠的临时区域；二是焦点还会在发布前过滤器中强制保留角色，超出了“只切换 constant”的约定。rc.25 已补齐：

- 自动区域必须具备稳定 `baseContent`、已固化历史，或离场后仍成立的明确变化；
- 仅进入、停留、提及、设施列举和路径说明不再生成区域对象；
- 当前路径上的设施/功能描述继续只保留在“时空”；
- 人工、完全锁定、独立稳定定义和封锁/损毁/占领等持续区域状态继续保留；
- 新增 `regionAllocationV28` 迁移位，旧聊天会幂等重新执行区域分配；
- 焦点不再改变参与发布的对象集合，只把已经入选的唯一角色从 `trigger` 切换为 `constant`。

新增 `rc25-comprehensive-audit` 回归，固定检查默认八视图、所有源码模块注释边界、区域分配、单一焦点仅常驻、同 ID 非角色不继承焦点，以及小总结/大总结稳定逻辑键。详细结果见 `RC25_COMPREHENSIVE_AUDIT_REPORT.md`。

## rc.24 实机问题修复

rc.24 针对 rc.23 实机发现的旧小总结残留、状态分配错误和刷新后世界书分裂进行修复：

- 同一 `event_id` 的小总结只保留一个活动版本并发布为一条世界书；旧版本保留历史但退出发布和大总结消费。
- rc.23 以前的多条未固化小总结会迁移为一个累计活动版本；缺少 `event_id` 的摘要不按标题猜测合并。
- 最新大总结使用固定世界书键 `large:current`。
- 当前地点的设施说明只进入“时空”，不再因为“设有、用于、结构、功能”等描述复制到“区域”。
- 区域只保留离场后仍成立的明确变化、人工内容和已固化历史。
- 对象跨轮继承稳定 ID，世界书同时使用物理键和逻辑键归并。
- 同一聊天的自动同步、手动同步和刷新补同步串行执行。
- 保存后以及世界书编辑器重绘后都会回读归并，处理旧前端缓存延迟回写。
- 共享世界书只清理可证明属于当前聊天的无元数据旧条目，明确属于其他聊天的条目继续保护。

详细边界见 `RC24_REAL_MACHINE_FIX_REPORT.md`。

## rc.23 对象视图升级

默认视图收拢为八类：

1. 时空
2. 角色
3. 物品
4. 事件
5. 区域
6. 全局变化
7. 基础设定
8. 自定义对象

“焦点”不再是事实表或世界书正文；“关系”和“技能”不再默认单列。角色对象内部统一保存：

- 基础内容
- 已固化历史
- 当前状态
- 关系状态
- 能力状态
- 关联对象
- 关联事件

区域、物品、事件、基础设定和玩家自定义对象同样使用“基础内容 + 已固化历史 + 当前状态”的结构。表格仍是动态注册视图，玩家可以新增、删除、重命名、调整用途与字段、排序、启停和恢复默认八表。

## 单一焦点常驻

每个聊天最多设置一个角色为焦点。焦点：

- 只能由玩家在角色视图中手动设置或取消；
- 只把该角色对应的世界书条目切换为 `constant`；
- 不写入条目正文；
- 不赋予额外权重、排序、容量保护或裁剪豁免；
- 不由状态模型、小总结或大总结创建和切换；
- 切换时不调用模型，只重新同步世界书。

即使关闭“最新连续性常驻”，玩家手动设置的焦点仍保持 `constant`。同 ID 的非角色对象不会继承焦点常驻。

## 基础层与可变状态

对象基础内容一旦存在，状态提取不得改写。基础层为空时，允许正文首次明确对象身份或固有定义后初始化。`solidifiedHistory` 仍只能由后续长期总结固化，普通状态提取不能新增。

玩家人工创建的普通对象默认采用“基础保护”：基础内容受保护，当前状态、关系状态、能力状态和关联信息可以依据明确事实更新。完全锁定对象的基础和状态均不允许自动修改。

状态写入必须有明确对象、事实来源和持续作用。以下情况不得自动赋予关系、能力或其他状态：

- 仅仅同场、出现、听见或围观；
- 普通对话或临时表情；
- 模型推测、身份联想或自动补全；
- 对后续没有持续作用的瞬时反应。

## 世界书身份与维护

普通同步的匹配顺序严格限制为：

1. 当前 `chatKey + managedKey` 唯一命中；
2. 当前 `chatKey + logicalKey` 新旧两侧均唯一；
3. 否则创建新条目。

同一 `managedKey` 的当前聊天托管副本会稳定保留一个并删除其余副本。`logicalKey` 多候选时不接管。owner 未知的 `[MA11]` 条目、其他插件条目和其他聊天条目不会在普通同步中接管、改写或删除。

备注、精确签名以及旧 `small:<id> / large:<id>` 考古已移到玩家主动的“清理并重新发布”流程。维护操作先展示统计并要求确认；共享世界书中无法证明 owner 的条目仍然保留。

## 旧数据迁移

首次读取旧聊天时自动执行对象视图迁移：

- `state / characters → 角色`
- `events → 事件`
- `regions → 区域`
- `foundations → 基础设定`
- `relationships → 角色.relationshipStates`
- `skills → 角色.abilityStates`
- 自动焦点视图退出，不作为事实迁移；人工或锁定焦点备注保留到自定义对象
- 无法明确归属的旧关系和技能进入“待归并”自定义对象
- 涉及三个以上角色、归属不唯一或名称匹配不明确时，不任意分发
- 已成功归并的人工/锁定旧关系和技能保留迁移凭据，但不重复发布世界书
- 玩家自定义视图及旧默认表上的自定义字段继续保留

旧人工行、锁定行、内部事实、小总结、大总结及其消费关系不会因视图迁移而删除。

## 内部事实与总结

聊天级事实层独立于表格，保存稳定 `fact_id`、`event_id`、来源消息、已发生事实、未解决事项、状态、时间范围、关联对象，以及小总结消费和大总结固化标记。

小总结仍按单一 `event_id` 汇总已经发生的事件线。大总结仍只读取上一版大总结和尚未固化的小总结。同一小总结不会重复固化。

rc.25 尚未实现“普通自定义条目自动归并到对应对象、总结向对象状态分发、多个事件影响综合和稳定事实写入已固化历史”的完整新机制；这些属于后续独立阶段，不在本次注释/审计版中伪装完成。

## 世界书召回

镜渊托管条目只使用：

- `constant`：玩家手动焦点、当前时空、必要规则和直接相关全局变化等既有常驻项；
- `trigger`：基于名称、别名和 `any / all / exclude` 条件触发；
- `vector`：未固化小总结和最新累计大总结。

`order` 只用于 SillyTavern 世界书编辑器显示，不参与记忆取舍。实际向量门槛与 Max Entries 仍由 SillyTavern Vector Storage 全局设置控制。

## 历史重建

- 正文与审核规则未变化时复用审核结果；
- 从第一个受影响状态开始重建，不执行无差别全链 force 重跑；
- 只重建受影响事件线的小总结；
- 大总结只消费新的未固化小总结；
- 世界书重新发布不调用模型；
- 编辑、删除、聊天切换、禁用、重启和重置都会使旧任务失去提交资格。
