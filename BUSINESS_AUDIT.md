# Mirror Abyss 1.2.0 稳定版业务逻辑审计

流水线：`ma-pipeline-60`


## 1.2.0 稳定版补充边界

小总结和大总结的模型规则由硬编码黑盒改为五段白盒配置：核心判断、保留内容、排除内容、更新规则和表达规则。用户可以调整“什么信息属于这一层”，但固定文本标签、事件槽位、消费关系、版本链、固化事务和世界书写入仍由插件代码控制。

小总结恒定维护近期事件连续性；大总结恒定维护跨场景、跨阶段仍不可遗忘的长期事实。两层不以固定字数区分，也不允许大总结退化为小总结的近义缩写。

表格普通设置只暴露表头名称与记录要求。稳定字段键、字段类型、保护权限和旧数据映射继续由插件维护；界面简化不得改变已有快照结构或历史数据身份。

## rc.59 异步与协议补充边界

聊天切换不仅取消模型 AbortController 和其他聊天 pending，还必须给当前 TaskQueue active job 留下永久取消标记。只比较 chatKey 无法识别 A → B → A；宿主保存或世界书 I/O 的迟到结果必须由既有 task guard 拒绝。

固定文本普通块必须用对应结束标签闭合。缺失结束标签、嵌套新块、错配结束标签或孤立结束标签属于协议失败，不得把已读字段作为半成品提交。`<MA_REPLACEMENT>` 原文块只识别自身结束标签，正文内出现其他协议标签仍作为原文保留。

## 本版核心边界

rc.58 不改变对象准入、事实数量、变化幅度或旧快照修订原则。变化只发生在模型输入的旧数据表达：完整内部快照继续由插件持有，模型只接收全局对象短目录与少量高相关工作副本。

`<MA_DIRECTORY>` 只提供表格、规范名称和少量别名，用于身份识别；它不是白名单。目录未列出的正文新对象仍可输出，插件在完整内部快照中再次按名称和别名查重后决定沿用旧 ID 或创建新 ID。

`<MA_CONTEXT_ROW>` 只包含直接命中对象、当前时空、活跃事件、当前角色及直接关联对象的精简当前切面。对象未进入工作副本但本轮确实变化时，模型仍可按目录名称返回变化字段；插件从完整旧快照保留未返回字段。

rc.57 的单项表达收紧与确定性差异抑制继续生效：不存在总条数硬上限；剧烈变化必须完整返回；空白、末尾标点和数组换序不构成真实更新。

rc.52 将“模型生成结构化数据”和“插件管理内部数据”彻底分离；rc.53 在此基础上把“同表同名唯一”提升为对象层、历史层和世界书出口共同遵守的硬约束。

模型通信不再使用 JSON。模型输入中的相关事实、旧对象索引和旧行全文由插件序列化为 `<MA_CONTEXT_…>` 固定文本块；模型输出使用任务对应的固定标签文本。模型不再返回 JSON、JSON Schema 对象、稳定 ID、整表快照或数据库补丁。

JSON 只保留为插件内部的设置、快照、任务、表格、迁移和诊断存储格式，不再作为模型通信协议。



## 历史恢复检查点边界（rc.56）

历史核心重建采用逐消息提交。每条状态提取成功后，其 artifact、内部事实、快照和处理凭证立即固化；后续消息失败不得回滚或重新调用已经成功的前缀。

`historyRecovery.currentIndex` 是下一条待处理或本次失败消息，`completedCount` 是当前可处理消息序列中已经成功固化的数量。再次执行恢复时，必须按 `currentIndex` 在当前可处理序列中的位置切分后缀，不得重新从 `historyInvalidation.startIndex` 开始。

只有再次发生历史编辑、换页或删除并触发新的历史失效时，旧检查点才作废。若核心阶段已完成而派生总结或世界书失败，重试直接进入派生恢复。


## 运行时队列边界（rc.55）

模型空响应、协议错误和上游失败必须进入终态并释放任务去重键。持久化阶段记录只用于展示和恢复，不得单独充当运行时互斥锁；`queued/running` 是否仍然有效，以当前 `TaskQueue` 中同聊天、同消息、同任务类别的真实活动记录为准。

玩家显式提交小总结或大总结时，手动任务可以接管同聊天的陈旧自动派生链：尚未执行的自动总结和自动同步被取消，正在运行的自动总结被请求取消。已开始的其他高优先级核心任务和世界书物理写入不被强制并发打断。

## 模型职责

模型只负责读取固定文本上下文，并填写固定文本壳中的自然语言内容：

- 审核：`<MA_AUDIT>`、`<MA_VIOLATION>`、`<MA_REPLACEMENT>`；
- 状态提取：`<MA_TURN>`、`<MA_FACT>`、`<MA_ROW>`；
- 小总结与大总结：`<MA_SUMMARY>`；
- 定向修正：返回完整修正正文，不包数据库结构。

模型不得：

- 生成或选择稳定 ID；
- 直接指定内部行键、事实键或事件键；
- 返回完整 JavaScript/JSON 对象；
- 决定覆盖数据库中的具体记录；
- 复制整张旧表；
- 通过未注册字段扩展内部结构。

固定标签和字段名是协议边界，字段内容允许使用自然语言。缺少必要标签、重复唯一槽位、出现未知槽位或无法闭合的关键结构时，插件拒绝提交，不猜测写入。

## 插件职责

插件是唯一的数据管理员，按以下顺序处理模型输出：

1. 解析固定标签块；
2. 规范化字段名、空白、分隔符和列表；
3. 根据表格、对象名、关键词、别名、事实和事件关联匹配已有对象；
4. 合并同一次响应中的重复条目；
5. 对有充分证据的历史重复行执行收拢；
6. 沿用已有稳定 ID，或由插件为真正的新对象生成 ID；
7. 应用人工锁、基础层保护、字段注册表和历史事务校验；
8. 转换为内部对象并写入聊天快照；
9. 在事务成功后进入总结、世界书同步和诊断链路。

任何模型文本都不得绕过解析器直接写入内部表格。

## 对象身份与去重边界

对象身份不依赖模型提供的 ID，并遵守以下硬约束：

> 同一聊天、同一表格、同一规范名称只能存在一个对象。

空白、常见分隔符、括号、书名号和大小写差异由全链共用的名称规范化函数处理。状态解析、快照归一化、保护层合并、手工编辑、历史迁移和世界书发布不得各自定义不同的“同名”算法。

不同事实、事件、时间和状态属于同一对象的不同切面，应进入 `currentFacts`、`currentStates`、`recentHistory`、`solidifiedHistory`、事实 ID、事件 ID 与其他注册字段，不得通过创建同名行表达。

插件按以下层级维护唯一身份：

1. 同一次模型响应中的同名 `<MA_ROW>` 先合并；
2. 同一快照中的同表同名行无条件归并；
3. 不同历史快照中的同一规范对象沿用首次稳定 ID；
4. 手工新增或改名命中同名对象时更新原对象，不新增第二行；
5. 关联对象引用和玩家焦点 ID 随旧 ID 映射改写；
6. 世界书待发布计划按完整托管名称执行最终唯一化；
7. 当前聊天已有同名托管副本确定性保留一个 UID，并删除其余副本。

人工基础保护和完全锁定仍决定合并后的内容优先级，但不赋予同名行共存权。真正需要区分的同名对象必须在首次建立时使用不同的规范名称，世界书不使用 ID 后缀临时掩盖身份冲突。

旧聊天通过一次性 `uniqueObjectNamesV31` 迁移扫描全部历史快照。迁移保存旧 ID 到规范 ID 的别名，即使聊天已保存而 metadata 保存失败，也能在刷新后继续修复焦点和引用，不会重新制造重复。

## 内部 JSON 边界

以下内容继续使用插件内部 JavaScript 对象和 JSON 序列化：

- 扩展设置与动态表格注册表；
- 聊天状态、事实账本和对象快照；
- 小总结、大总结和消费记录；
- 任务队列、请求轨迹和诊断导出；
- 世界书逻辑键、物理键和同步映射；
- 版本迁移与历史恢复检查点。

这些 JSON 全部由插件代码产生、校验和迁移。模型无法直接构造或提交这些对象。

## 已移除的旧模型协议设施

rc.52 运行时不再包含：

- OpenAI `response_format` / `json_schema` 请求；
- 通过 `JSON.stringify` 向模型传递内部事实、索引或旧行；
- strict JSON Schema 模型输出；
- Schema 能力缓存与无 Schema 回退；
- “兼容模式”请求开关；
- 模型 JSON 提取、代码块清理和 JSON 修复重试；
- 由模型生成稳定 ID 或完整表格补丁的路径。

旧设置键仅在加载旧配置时被删除，用于迁移清理，不会重新启用旧协议。

## 人工数据与基础层保护

固定文本协议不改变既有数据保护边界：

- 人工锁定行不得被自动流程删除或覆盖；
- 非空 `baseContent` 原则上不由普通状态提取改写；
- 明确重定义、毁灭、重建或人工编辑才可改变基础定义；
- 未返回的旧行继续保留，空响应不等于清空整表；
- 未注册字段不会进入内部对象；
- 焦点只影响世界书常驻状态，不改变对象身份、排序或裁剪规则。

## 历史重建事务边界

历史失效后，普通状态整理、总结与世界书同步继续受锁约束。

重建过程必须：

- 使用同一聊天范围快照；
- 每次提交前验证聊天、历史修订和消息指纹；
- 失败时保留最后已确认快照和恢复检查点；
- 不把未完成重建的数据发布到世界书；
- 只有重建、发布和必要回读全部成功后才清除历史锁。

固定文本解析失败属于当前消息处理失败，不允许退回旧 JSON 协议，也不允许将原始文本当成有效表格数据提交。

## 世界书同步边界

世界书仍是长期记忆发布层，不是模型输出的直接落点。

同步只消费已经通过插件解析、去重、校验并提交的内部状态。逻辑键、物理键、聊天归属、人工条目保护、保存后回读和重复归并规则继续生效。

## 请求与诊断边界

当前聊天连接和 Connection Profile 均发送普通文本请求，不附带 JSON Schema 或结构化输出字段。

诊断记录协议类型、连接 lane、任务类型、耗时、状态和错误分类，但不记录玩家正文、完整模型输出或 API 密钥。连接成功只证明对应实测请求成功，不以服务对象存在代替真实请求结果。

## 验收结论

rc.53 保留 rc.52 的固定文本模型协议，并完成对象唯一性闭环：模型负责内容理解，插件负责身份、切面合并、分发、稳定 ID、历史迁移和世界书唯一发布。相同规范名称不会再以多个表格行或多个当前聊天世界书条目存在。
