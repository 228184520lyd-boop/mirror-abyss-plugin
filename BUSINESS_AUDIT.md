# Mirror Abyss 1.2.0-rc.31 业务逻辑审计

流水线：`ma-pipeline-32`


## rc.31 行级补丁边界

rc.31 将状态传输从“变化整表”改为“变化行补丁”。非空数组只更新返回行，未返回旧行继续保留；只有显式空数组才清空整表。模型上下文只携带稳定 ID 索引、相关旧行全文与相关事实，避免每轮重复发送完整旧快照。Schema 不兼容缓存按连接共享并保存在当前浏览器会话中。

## rc.30 实机恢复链边界

rc.30 不改审核、总结、世界书发布内容与对象规则，只修结构化状态传输和历史恢复可观测性：snapshot 允许只返回变化表格，未返回表格继承旧值；输出截断不得进入模型修理；失败不得覆盖旧快照；历史恢复记录明确阶段与错误。

## 基线与修改范围

rc.29 以正式 rc.28 源码为唯一基线，继续保留审核、定向修正、状态提取、小总结、大总结、世界书同步、历史重建、聊天隔离、任务队列、请求 lane、提交守卫、Connection Profile、诊断和自动流程。本版只让默认语义表头成为可编辑模板，并补注册表提交守卫；不增加对象字段、迁移位、叙事能力或总结分发。

## rc.29 语义表头边界

- 玩家可修改 `title / content / status / keywords` 的 label 和 description。
- 玩家 label 进入状态提示词、JSON Schema 描述、表格主列、对象编辑器和世界书正文；description 进入提示词、Schema 与表格管理编辑区。
- `table.key / role / field.key / type / required / row.id` 永久保持机器身份，不随改名变化。
- 默认表头未变化时，世界书正文维持 rc.28 格式，避免无意义重写。
- 状态请求发出后若启用注册表 fingerprint 变化，旧结果在写入事实包和快照前拒绝提交，阶段恢复为可重试。
- 自定义字段类型不可直接修改；同一次“删旧键 + 加新键”不作为改表头处理。
- 表格定义仍是全局扩展设置，适用于所有聊天；没有新增迁移位或 chat state 字段。

## rc.28 核心稳定边界

- 焦点只校验当前快照现有角色，不跨历史 artifact、名称或旧 ID 推断。
- 状态缓存指纹基于完整规范化请求；稳定身份回退要求新旧两侧均唯一。
- 三层旁观过滤共享一个领域谓词。
- create/sync/pause/clear 以最终物理世界书名共享 mutation gate。
- 普通同步只按当前聊天 `managedKey`、唯一 `logicalKey` 匹配；历史考古进入玩家主动维护。
- 当前 schema 读取绕过全部历史 artifact 扫描和旧迁移规范化；旧 schema 保留原 V23–V30 路径。
- 当前写入不再无条件重跑旧总结消费迁移，事件线新事实可以产生下一版小总结。

这些变化没有删除 V23–V30、TaskQueue、RequestLaneScheduler、人工/锁定保护、base/current 分离或保存后必要确认。

## rc.25 注释与审计边界

- 39 个 TypeScript 源模块全部具备“模块职责/维护边界”头注释。
- 关键内联注释覆盖旧表迁移、待归并保护、对象稳定 ID、基础层保护、人工 base/all 锁、摘要版本事务、世界书物理/逻辑身份、共享书删除证明和三次回读归并。
- 注释解释约束原因，不复述简单赋值、DOM 操作或显而易见条件。
- `focus/state/skills/relationships` 旧角色仍保留在类型与迁移层，仅为读取旧设置/快照；默认运行时八视图不重新创建它们。
- rc.24 区域分配只处理当前路径镜像的问题已补全：无独立基础、无长期历史、无持续变化的自动区域一律退出，不再因为“另一个被提及地点”绕过过滤。
- 焦点从发布前过滤器中移除；设置焦点前后对象集合必须一致，只允许目标角色的 `recallMode` 发生 `trigger → constant` 变化。
- `regionAllocationV28` 保证旧聊天重新执行一次新的幂等分配。

## rc.24 实机修复边界（继续保留）

- 小总结以 `event_id` 为活动版本身份，世界书固定使用 `small:event:<event_id>`；旧版本保留但退出发布和大总结消费。
- 大总结固定使用 `large:current`，版本更新只覆盖同一条目。
- 当前地点描述只进入时空；区域只接受人工内容、已固化历史或离场后仍成立的明确区域变化。
- 对象 ID 跨快照继承；世界书按物理键、逻辑键、完整备注和精确签名复用。
- 同聊天同步串行，保存后和编辑器重绘后均回读归并。
- 共享世界书不触碰明确属于其他 chatKey 的条目。

## 对象化可见视图

默认视图由 rc.22 十表收拢为八类：时空、角色、物品、事件、区域、全局变化、基础设定、自定义对象。

表格仍是动态视图，不是唯一数据源。删除或停用视图不会删除：

- 聊天级内部事实；
- `fact_id / event_id`；
- 小总结和大总结；
- 历史重建依据；
- 玩家人工世界书条目。

### 角色对象

关系和技能不再默认单列，而是进入角色对象：

- `baseContent`
- `solidifiedHistory`
- `currentStates`
- `relationshipStates`
- `abilityStates`
- `relatedObjects`
- `relatedEvents`

关系状态必须由明确关系变化支持；能力状态必须由明确使用、获得、失去、受限或改变支持。同场、普通对话、表情、推测或身份联想不能自动赋予。

### 其他对象

物品、区域、事件、基础设定、全局变化和自定义对象使用同样的基础层、已固化历史和当前状态层。玩家新增普通对象默认基础保护；完全锁定对象保持全字段保护。

## 基础层写入边界

状态提取只允许：

- 对新对象写入正文首次明确的稳定基础；
- 更新当前状态、关系状态、能力状态和关联；
- 保留已有基础内容与已固化历史。

一旦 `baseContent` 非空，后续状态模型不能改写。`solidifiedHistory` 不属于状态提取写区；旧值原样保留，模型新增值被丢弃。

## 旧数据迁移

### 无歧义映射

- `state / characters → characters`
- `events → events`
- `regions → regions`
- `foundations → foundations`
- 其他同名对象视图直接沿用

### 关系与技能归并

旧关系根据角色稳定名称归并：

- 明确命中一个或两个角色时写入对应 `relationshipStates`；
- 命中三个以上、没有命中或存在歧义时转入自定义对象并标记“待归并”；
- 不允许任取前两个角色。

旧技能只有唯一角色归属时写入 `abilityStates`，否则转为待归并。

人工或锁定旧关系/技能成功归并后，原记录保留为“已归并”迁移凭据，继续受人工保护，但不会再次发布世界书，因此不产生内容副本。

自动焦点行不属于事实，不迁移。人工或锁定焦点备注保留为自定义对象。

### rc.27 已持久化角色

`characterMergeV29` 继续负责尚保留双表、相同稳定 ID 的确定性合并。独立 `persistedCharacterMergeV30` 只负责 rc.25 已经保存成单个角色表、且状态行 ID 已重新生成的形态。

V30 不做一般标题去重。只有同一规范名恰好存在一条受保护基础行和一条无保护状态行时才合并；多候选、占位名、近似名和冲突保护全部保留。合并同时重映射焦点及精确对象引用，并在世界书重新同步前把旧成功状态改为待同步。

V30 不触发区域分配或全表重建，未注册旧表保持原样。聊天保存失败回滚，metadata 保存中断通过随 artifact 保存的 ID 别名续迁；同一聊天并发读取只执行一次迁移。

## 单一焦点常驻

每个聊天在 `ChatState.focusObjectId` 中最多保存一个玩家手动选择的角色。

焦点仅产生一项业务效果：

```text
该角色世界书条目 recallMode = constant
```

明确不产生：

- 额外排序；
- 权重；
- 容量锁定；
- 裁剪豁免；
- 召回优先分；
- 正文中的“玩家控制”描述；
- 模型自动设置或切换。

焦点判断同时校验表格角色，只对 `characters / state` 生效。其他表出现相同 row ID 也不会常驻。玩家切换焦点时只更新当前聊天 metadata 并强制重新同步世界书，不调用模型。

## 世界书边界

对象文档物理稳定键仍为：

```text
view:<tableKey>:<rowId>
```

焦点只改变对应角色文档的 `constant`。其他世界书模式继续由既有业务规则和 SillyTavern 触发/向量机制处理。

“已归并”的人工迁移凭据只保存在镜渊 UI 和聊天数据中，不发布为世界书条目；普通未归并人工对象仍正常发布并受保护。

普通同步不再按备注、精确签名或旧 `small:<id>/large:<id>` 考古。只有玩家主动执行“清理并重新发布”时才预览并处理可证明属于当前聊天的旧托管副本；共享书 owner 未知条目继续保留。

## 总结与事实层保持

内部事实层和历史重建边界保持；小总结新增活动版本链与旧版迁移，大总结消费时排除已被替代版本。

本版以及 rc.25 仍没有实现：

- 小总结向多个对象状态分发；
- 大总结向对象已固化历史分发；
- 多事件影响的独立 add/update/close/replace 操作层；
- 多事件状态综合与冲突队列。

这些能力将在对象视图完成实机确认后作为后续独立阶段开发，避免和基础迁移同时引入大范围不可验证变化。

## 历史重建与提交边界

- 正文和审核规则未变化时复用审核；
- 只从受影响状态开始重建；
- 核心重建期间跳过逐条派生；
- 小总结只重建受影响 event_id；
- 大总结只消费未固化小总结；
- 世界书重新发布不调用模型；
- 所有写入继续受任务代次、历史修订号、chatKey、消息身份和正文指纹守卫。

## 明确保留的边界

- 不生成或拦截 SillyTavern 主叙事正文。
- 不保存独立 API 地址、密钥或供应商配置。
- 不切换 SillyTavern 全局 Connection Profile。
- 不修改 Vector Storage 全局门槛或 Max Entries。
- 不引入业务并发。
- 不删除人工世界书条目。
- 不把表格重新提升为事实真相来源。
