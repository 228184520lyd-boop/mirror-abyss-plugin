# Mirror Abyss 业务逻辑审计

## 保持的核心链路

1. AI 正文进入规则审核。
2. 审核通过，或定向修正并复审通过后，正文才进入记忆链。
3. 单次事实与状态任务先输出标准化事实，再输出九类当前状态表。
4. 标准化事实附着在对应消息上，状态表作为当前快照附着在消息上。
5. 小总结消费未总结的标准化事实，并进行受限制的安全沉降。
6. 大总结合并若干小总结和上一版大总结。
7. 最新状态、当前小总结和累计大总结发布到聊天世界书。
8. 编辑、切换 swipe 或删除旧消息后暂停派生处理和发布，由用户从最早变化点重算。

## 有意简化

- 不保存独立 API 地址或密钥。
- 不维护事实包数据库、提交日志、跨标签协调或恢复事务。
- 不自动锁定生成按钮，不拦截酒馆生成。
- 不自动删除审核失败的消息。
- 历史重算不在后台自动开始。
- 每个模型任务一次正常调用；结构解析失败最多一次显式格式修复。

## 与旧 rc.4.8 的实现差异

- 旧版事实提取可跨多条消息自动批处理；当前版按单条消息处理。
- 旧版可缓存并复用事实包；当前版历史重算重新调用模型。
- 旧版有独立事件登记和长期候选记录；当前版由事实包、状态表和总结直接承载。
- 旧版包含世界书 outbox、回读验证和冲突处理；当前版直接使用 SillyTavern 官方世界书读写接口。

这些差异降低了复杂度和隐性行为，但没有删除审核、事实标准化、状态维护、分层总结、沉降和世界书发布等产品能力。

## 测试覆盖

- 事实字段规范化及无效事实过滤。
- 状态表规范化和稳定 ID 连续性。
- 已结束事件允许沉降，基础设定禁止沉降。
- 人物活动状态只允许逐级降为休眠。
- 剧情密集提前总结、闲聊延后、达到上限强制总结。
- 删除中间消息后的最早不一致位置识别。
- manifest、设置入口、生命周期、原生模型调用和世界书接口静态检查。
- 小总结状态表与聊天索引保存失败时回滚。
