# Mirror Abyss 1.2.0-rc.17 业务逻辑审计

## 保留的核心链路

1. 只有明确的 AI/角色正文进入自动处理；玩家、系统、空白和来源不明消息不触发。
2. 正文先审核。审核通过，或定向修正并复审通过后，才进入记忆链。
3. 单次状态任务同时产出标准化事实包和九类当前状态表。
4. 事实与状态先附着到对应消息并写入聊天元数据。
5. 小总结消费尚未总结的事实，并执行受限沉降。
6. 大总结消费尚未合并的小总结。
7. 最新状态、总结和连续性文档发布到本聊天世界书。
8. 旧消息编辑、swipe 或删除后取消旧排队任务、暂停本聊天世界书，并由用户选择起点重算。

## 本次重构

原版把审核、状态、小总结、大总结和世界书全部放在一个长任务中。任何低优先级总结一旦排在前面，后续正文都只能等待。

当前版仍只有一个插件执行器，但改为优先队列，并把任务拆为：

- 关键链：审核、修正、状态、事实提交。
- 派生链：世界书发布、小总结、大总结。

关键链优先于所有排队中的派生任务。活动中的请求不做危险的强制抢占；聊天切换、历史变化、禁用和重置时使用原生 `AbortController` 取消并阻止旧结果提交。

模型请求另有按连接通道串行的轻量调度器，用于记录连接等待和请求耗时，并防止同一 Profile 被插件并发轰炸。

## 有意不做

- 不新增独立 API 客户端、密钥存储或供应商适配层。
- 不切换 SillyTavern 全局 Connection Profile。
- 不开放无界并发，不把插件改造成工作流框架。
- 不对活动中的低优先级请求做不可靠抢占。
- 不强开流式输出；结构化结果仍完整返回后统一解析。
- 不增加事务数据库、outbox、后台自动历史重放或跨标签协调。
- 不拦截酒馆生成，不自动删除审核失败消息。

这些取舍优先解决真实的排队、取消和误触发问题，同时控制复杂度。

## 数据安全边界

- 每次提交前校验当前聊天、消息位置与正文指纹。
- 聊天切换后取消其他聊天的排队任务。
- 历史变化后取消当前聊天旧任务，并暂停仅属于该 `chatKey` 的世界书条目。
- 诊断记录只包含任务类型、连接描述、阶段、优先级和耗时，不包含提示词、正文、响应全文或密钥。


## rc.12 定向修正恢复

- 旧版 `withdraw` 设置升级后恢复为“定向修正并原位替换”，不再静默迁移成隐藏。
- 审核结果即使为 `block`，在用户选择自动修正时也先进行一次有上限的定向修正，不再直接跳到人工撤回/隐藏。
- 修正进行中保留原正文并标记，不先隐藏；候选正文复审通过后才原位替换活动 swipe。
- 修正失败仍按用户选择标红或隐藏，不自动删除酒馆消息。
- 未提交候选的复审结果只写入修正历史，不覆盖当前正文的审核结果；只有候选通过并替换后才更新最终审核。
- 简易扩展设置中直接显示审核失败处理方式，避免功能藏在工作区深层。


## rc.13 修正进度可见性

- rc.12 的定向修正确实会启动，但阶段切换到 `running` 后只更新了内存对象，没有持久化聊天或触发流水线 UI 通知。
- rc.13 在修正模型请求发出前发布 `revision=running`：先保存当前 artifact，再通知消息面板和已打开的工作区。
- 该修复不改变审核决策、修正提示词、请求数量、优先级或原位替换语义。
- 修正完成或失败后仍由原有提交链发布最终状态。


## rc.14 人工整理修正链

- “整理最新正文”与“重试定向修正”属于明确的人工修正操作；审核失败时强制定向修正，不受历史保存的 `mark` / `hide` 失败动作阻断。
- 自动接收正文仍尊重 `auditFailAction`，没有把人工覆盖扩散到自动模式。
- 自动关键链、普通强制重算和人工定向修正使用不同任务模式键；人工操作不会再被已经运行的自动任务去重并吞掉。
- 不开放同消息并发：两种任务仍由全局单执行器顺序执行，提交前继续检查聊天、消息身份与正文指纹。
- 修正失败后仍使用 `revisionFallbackAction`，不会删除消息。

## rc.15 最新正文重新生成边界

- `MESSAGE_SWIPED` 现在区分两类行为：最后一条且仍为当前最新 AI 正文时，视为“当前回复版本替换”；更早位置的 swipe 仍视为历史变化。
- 当前回复版本替换只删除该消息旧指纹对应的 artifact，从 `processedMessageKeys` 移除旧 key，删除直接消费该 key 的小总结及依赖这些小总结的大总结，并把 `latestSnapshotMessageKey` 回退到上一份成功快照。
- 当前回复版本替换不会写入 `historyInvalidation`，不会要求整段历史重算，也不会全局中断已经针对新指纹运行的人工修正。
- 只取消基于旧版本排队的派生任务和明确指向旧 messageKey 的流水线任务；更早消息的独立流水线任务与当前新指纹任务保留。
- 若 rc.14 已把最后一条 swipe 误写为历史失效，人工“整理最新正文/重试定向修正”会在确认该失效只指向当前最新 swipe 后清除标记，再继续审核、修正、状态与派生链。
- 定向修正入口增加提交守卫：进入修正和每次尝试前再次校验聊天与正文指纹，旧版本请求不能给新正文加标记或执行替换。

## rc.16 末端正文删除边界

- SillyTavern 的 `MESSAGE_DELETED` 在消息删除后只传入新的 `chat.length`，不提供真正的删除位置。镜渊不再把“无法从事件参数定位”直接等同于全局历史损坏。
- 删除后若剩余消息上的镜渊 artifact 出现消息索引、身份或正文指纹错位，仍按旧历史删除处理：取消旧任务、暂停世界书并要求从检测点重算。
- 删除后若所有剩余 artifact 仍一致，视为删除发生在镜渊已管理历史的末端：取消正在处理的消失正文，清除不再存在的 processed key、快照指针及其直接小总结/依赖大总结，不创建 `historyInvalidation`。
- 整理中的末端正文被删除时，活动审核/修正请求通过 `AbortController` 取消；即使供应商迟到返回，提交守卫也会因消息已不存在或指纹变化拒绝旧结果。
- 删除事件异步清理期间若新正文已经重新生成并完成状态提交，清理会重新读取当前 artifact，不回退新快照，也不再次暂停已恢复的派生链。
- rc.15 遗留的 `deleted` 且无 `startIndex` 假失效，会在人工整理新正文前进行 orphan key 对账并安全清除；有明确起点的真实历史失效不会被自动解除。


## rc.17 UI 与业务边界收拢

- 审核与定向修正不再查询消息 DOM 或修改 class；它们只发布 `AuditPresentationState`。消息面板订阅该事件并负责隐藏/标记。没有 UI 订阅者时，后台审核、修正与提交仍正常运行。
- 简易设置和工作区的启停动作统一调用 `setPluginEnabled()`；请求接收、任务队列接收、活动请求取消和设置保存不再由两个 UI 分别拼装。
- 手工状态行保存与删除统一调用 `saveManualStateRow()` / `removeManualStateRow()`；提交守卫、消息 artifact 绑定、聊天保存及可选世界书同步由服务层负责。
- `bootstrap/app.ts` 仍作为组合入口连接 UI 与业务，这是有意保留的正常依赖方向；`core/domain/llm/pipeline/storage/services` 不导入 `ui`。
- 未引入状态管理框架、MVC、Redux、额外数据库或新事件总线；新增的展示事件只服务审核可见性，保持单向通知。
