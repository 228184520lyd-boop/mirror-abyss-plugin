# Mirror Abyss 1.3.15 业务逻辑审计

流水线：`ma-pipeline-78`

## 1.3.15 自然模块、分发与宿主边界

模型只在 `<MA_EVENT>` 围栏内输出稀疏自然事实。事件核心按需出现；对象模块第一行声明对象，事实行直接写该对象发生的变化。不同对象的不同变化可以分别存在，同一对象的同一变化只能出现一次。事件没有独立过程事实时，插件只维护事件名称、状态与关联对象，不再要求模型复述整体动作。模型不决定数据库字段、稳定 ID、事实 ID、事件 ID、宿主选择、生命周期或删除。

事件、临时 NPC、场景、普通终态物品、小总结和大总结均可作为加工容器，而非天然永久世界书正文。只有持续结果已经分发到稳定宿主、没有未决事项、事实已被对应总结覆盖时，统一状态机才允许 `active → settling → 物理删除`。无对象分发、覆盖不完整、焦点、人工、锁定或持续角色都不能被自动清除。

小总结和大总结的精度必须逐层下降。小总结保留近期结果与未决影响，大总结只保留跨阶段仍成立的长期变化；它们不得把原事件摘要复制到所有对象。总结正文完成分发后可从处理列表退出，只保留不参与召回的来源范围、消费事实和分发目标凭证。

基础定义是当前事实，不是不可改变的永恒常量。只有明确改变对象本质的事件模块可更新当前基础定义，同时保留稳定 ID 和变化证据；临时伪装、幻术、服装、受伤或阶段状态只能进入外观／当前状态。完全锁定对象继续阻止自动更新。

## 1.3.15 ST 接管与记忆网络边界

统一 ST 接管模块是托管世界书发布参数的唯一外围入口。插件启用后默认管理世界书、向量和递归；规则审核、向量和递归可以分别关闭。关闭某个外围能力只停止该能力，不得跳过状态事实、总结窗口或快照提交。

记忆网络是当前快照的只读投影。节点布局由焦点／当前事件与关系距离决定，不再按对象类型写死同心圆环；节点位置、统计和详情不能反向成为事实证据。手机端卡片和图谱尺寸只改变展示，不改变世界书正文、对象身份或生命周期。

## 1.3.13 模型协议与存储边界

模型只描述正文事实的语义位置，不操作数据库结构。生产输出限定为 `<MA_TURN>` 与 `<MA_CHANGE>`；“对象类型、对象、变化层、动作、内容”是模型协议，“baseContent / currentStates / stable id”等仅是插件内部实现。

`src/domain/state-semantics.ts` 是语义层到内部字段的唯一映射入口。固定中文层、自定义字段显示名和表格显示名可以进入模型上下文；内部字段键、表格键、稳定 ID、事实 ID、事件 ID、生命周期与结算指令不得进入模型输出。未知语义层必须拒绝，不能通过猜测或自动扩表写入。

旧 `<MA_FACT>`、`<MA_ROW>`、`field=`、`entry_action` 和 `merge_*` 不再作为兼容输入。对象身份、跨表纠正、稳定 ID、事实生成、生命周期和总结覆盖仍由既有确定性状态机完成，因此本版改变的是通信边界，不改变事实准入和记忆生命周期规则。

UI 表头、模型语义名称和内部稳定键是三个独立层。玩家可以改显示名称或自定义语义说明，但不能通过表头改名改变内部数据身份；默认语义层不会因 UI 文案变化而漂移。

## 1.3.12 历史工作流边界

历史失效与恢复流程由 `src/workflow/history-workflow.ts` 统一解释和推进。`pipeline.ts`、世界书发布与诊断只能通过该接口判断流程是否阻断、当前阶段、恢复起点和错误；不得各自维护一套历史状态转移。

本阶段底层仍保存 `historyInvalidation / historyRecovery`，但它们仅是工作流模块的存储实现，不再是外围业务可自由修改的两个权威。事实、快照、总结和世界书内容仍由既有记忆状态机与提交边界负责。

## 1.3.11 单一记忆权威边界

生产链只有一个记忆写入编排者：`memory-state-machine.ts`。模型提供明确事实，`object-identity.ts` 只解析稳定对象，`special-table-rules.ts` 只处理时空等确定例外，`entry-lifecycle.ts` 只执行参与状态与物理删除。总结、世界书、图谱和 UI 均不能自行创造另一套生命周期判断。

生产状态只允许 `active / settling`。旧 `absorbed / retired` 不进入正常游玩路径，只能由 repository 在加载旧存档时一次性迁移。模型输出中的 `entry_action`、`merge_*` 等字段不构成事实，生产解析器忽略。

小总结和大总结负责写入“事实已消费／已固化”的覆盖结果；随后统一状态机执行审计。世界书只读取最终提交态并映射为启用、停用或删除；旁观者过滤属于只读发布投影，不能写回快照。记忆网络同样只读。

旧 `sedimentation.ts` 与 `summary-policy.ts` 已删除，防止旧沉降、旧时机策略和新状态机同时成为权威。提交态通过不变量检查阻止重复稳定 ID 与旧生命周期状态重新进入生产快照。

## 1.3.7 确定性结算与状态传输边界

模型仍然只做无观点事实提取。主状态协议收敛为 `<MA_CHANGE>`：每个块表达一个正文明确建立的对象字段变化。模型不生成稳定 ID、事实 ID、事件 ID、生命周期、宿主选择、合并、删除或重要性判断。插件根据完整旧快照和注册表把变化投影为内部事实与对象补丁。

总结覆盖审计的时序属于插件事务边界。小总结必须先标记相关事实已消费，大总结必须先标记事实和小总结已固化，之后才能判断待结算容器是否已经被更高层承接。事件关闭时只沿 event_id 扩展本事件关联对象，不把全仓正文交给模型。

当前时空是聊天级单例，不是普通可重复对象。固定稳定 ID `spacetime_current` 表示“当前时间与位置槽位”；地点变化更新该槽位。旧存档中多个时空行由迁移器确定性合并并改写引用，避免新旧存档产生不同语义。

慢网关容错不得降低数据边界：标签围栏、空格、末块安全闭合等结构问题可本地规范化或进行一次紧凑修复；未知字段、禁用表、对象歧义、越权生命周期等语义错误必须拒绝。504 回退只拆分本轮正文，所有片段成功后才原子提交，不允许半轮事实进入快照或世界书。

状态上下文字符上限、输出 token 和分段字符数为白盒运行参数，但它们不能改变事实准入、ID、对象唯一性和事务提交规则。

## 1.3.6 书记模型与确定性管理边界

模型职责被压缩为事实转换：它只能返回正文明确建立的变化及对象投影，不能评价未来价值、推断伏笔、解释动机、枚举未知、判断事件结算或提出合并／删除。即使旧模型仍输出生命周期标签，生产流水线也忽略该部分。

插件在每轮提交后只处理本轮触及对象及其明确 event_id 关联，依据固定类型规则、显式终态、保护标记和总结覆盖执行参与状态迁移。该过程不需要把完整仓库正文发送给模型。角色自动退出保持保守；普通物品只有同时满足显式终态和普通／可替代条件才可自动结算，唯一、任务、证物和核心物品继续独立。

事件画像是派生索引，不是新的长期记忆层。硬字数上限是白盒发布边界：对象世界书内容按完整语义行保留，低优先历史层先退出；总结超限拒绝提交。内部事实账本不因展示上限被无规则截断。

## 1.3.5 世界书结算边界

世界书条目退出不再等于把对象搬入插件的永久隐藏墓地。模型只能提出 `absorb / retire / restore` 语义动作；插件负责对象匹配、保护校验、事实分发、世界书参与状态、总结覆盖审计和物理删除。

`absorb / retire` 首先进入短期 `settling`。该行仍保留在当前工作快照中以完成事务与同步，但发布到 ST 时被设为 `disable=true`，并清空常驻、关键词和向量参与资格。它不是长期记忆层，也不能无限滞留。

并入必须先找到唯一宿主并写入持续结果、事实/事件引用和必要召回名称。对应事件结束且小总结或大总结覆盖相关事实后，覆盖审计才允许删除源容器。事件未结束、事实未覆盖、宿主分发不完整、仍有未决事实，或对象属于人工、锁定、基础保护、焦点时，均不得删除。

允许级联结算：普通物品或一次性参与者可并入交易事件；交易事件的结果进入小总结后，源对象和事件容器可在同一轮结算中一起删除。删除的是工作容器，持续事实已经由活跃对象、宿主或总结承接。

1.3.4 的 `absorbed / retired` 仅作为旧数据兼容状态。插件在后续维护时检查总结消费、事实活跃性和宿主承接，满足条件后直接清除；新版本不再生成这类永久墓碑。

当前架构仍在聊天快照中保存表格与短期结算元数据，再差异发布到世界书；本版没有宣称把 ST 世界书改造成数据库唯一物理存储。已完成的是取消永久墓碑语义，并让退出最终落实为托管世界书与快照中的真实删除。

## 1.3.4 条目生命周期边界（历史行为）

独立条目是一种召回资格，不是对象一经创建便永久占用的数据库席位。自动对象失去独立价值后，插件允许模型提出三种确定动作：并入宿主、退出独立记忆、恢复独立记忆。模型只提出语义动作和归并事实；插件仍负责源对象匹配、宿主唯一性、稳定 ID、保护检查、证据继承与世界书差异同步。

并入时，源条目不会物理删除，而是成为隐藏墓碑。墓碑不进入普通表格、图谱、对象计数和世界书，但继续出现在模型的压缩身份目录中，防止相同名称再次被当成新对象。宿主吸收必要结果、事实/事件引用以及源名称和关键词召回；因此旧对象被提及时，召回的是承接其后果的宿主，而不是一个已经失效的独立条目。

退出仅适用于明确终态且没有值得承接后果的自动条目。出售或转移本身不自动退出：唯一物品、证物、任务物品、仍可追踪物品或其新所有权仍影响未来时，继续独立更新；普通可替代物品且出售结果已进入交易事件或资产变化时，才适合并入。人工、锁定、基础保护和焦点对象始终由玩家控制，不能自动归并或退出。

恢复必须显式发生。普通正文再次提到墓碑对象不会自动重新占位；只有其重新获得独立持续价值并返回 `restore` 时，插件才清除墓碑状态。

## 1.3.3 对象覆盖与语义分流边界

物品覆盖与角色准入采用不同门槛。背景人物仍需持续因果或关系资格；可区分物品只要其所有权、位置、数量、完整性、可用性、隐藏状态、用途或后续可取得性会影响下一轮，就可建档。插件要求逐件输出，不能把多件独立物品压成一个摘要。

场景、地点与时空是三个层次：时空表示当前时间和所在位置；场景表示本次局面切片；地点表示离开当前场景后仍成立的空间本体。场景表缺失时升级设置自动补齐；当前时空存在而模型漏写场景时，插件只生成最小场景条目，不凭空补写冲突或参与者。

模型通过 `kind` 声明对象本质，插件校验 table。语义纠正只自动迁移未锁定的自动条目，并保留稳定 ID、通用字段与引用；人工、基础保护和完全锁定条目不被自动搬动。目标表不支持的角色生命周期、关系与能力字段不会带入全局条目。

## 1.3.0 标准工作台边界

本版将“可调规则”与“不可由提示词接管的确定性边界”重新分层。用户可以编辑对象准入、背景排除、证据等级、冲突更新、两级总结、表格用途和表头记录要求；用户不能通过这些文本修改固定标签协议、稳定 ID、对象合并、事实消费、历史事务、聊天隔离或世界书删除安全。

对象提取允许显式放弃写入。身份不清、证据不足、同名无法确认、变化不成立或持续价值不足时，应省略条目或字段，而不是为了填满表格强制输出。传闻和主观判断不得自动升级为 confirmed。

关系图谱只从当前快照派生。显式 `relationshipStates`、`relatedObjects` 和 `relatedEvents` 优先；只在旧记录完全缺少显式关联时使用文本提及回退。图谱边不能反向成为事实证据，也不能触发对象创建或合并。

普通 UI 使用玩家任务语言并始终展示关键规则；实现协议集中到诊断页只读预览。无折叠不等于把所有底层结构铺在页面上：信息分层由页面职责、引导卡、搜索、状态反馈和诊断隔离完成。

## 1.2.1 对象准入白盒边界

“对象是否有资格建档”从状态系统提示词的固定黑盒规则中拆出，改为全局两段白盒配置：允许建档条件、默认排除条件。它只决定新对象是否可以输出 `<MA_ROW>`；每张表的用途说明与字段描述继续决定合格对象应记录什么。

默认策略采用保守准入：服装、外貌、站位、动作、短台词、姓名或当前场景出现均不能单独构成建档依据。路人、宾客、观众、侍卫、仆从、店员、士兵、村民、围观者及其他一次性功能角色默认排除。对象只有形成持续身份、独立因果作用、长期关系、状态、目标、责任或未决事项后才具备建档资格。

白盒规则不能接管固定文本协议、对象 ID、数据库匹配、合并、去重、历史继承或世界书发布。背景对象后续被正文明确升级为持续角色时，允许在该轮正常建档；插件不永久封禁某个称谓或职业。


## 1.2.0 稳定版补充边界

小总结和大总结的模型规则由硬编码黑盒改为五段白盒配置：核心判断、保留内容、排除内容、更新规则和表达规则。用户可以调整“什么信息属于这一层”，但固定文本标签、事件槽位、消费关系、版本链、固化事务和世界书写入仍由插件代码控制。

小总结恒定维护近期事件连续性；大总结恒定维护跨场景、跨阶段仍不可遗忘的长期事实。两层不以固定字数区分，也不允许大总结退化为小总结的近义缩写。

表格普通设置只暴露表头名称与记录要求。稳定字段键、字段类型、保护权限和旧数据映射继续由插件维护；界面简化不得改变已有快照结构或历史数据身份。

## rc.59 异步与协议补充边界

聊天切换不仅取消模型 AbortController 和其他聊天 pending，还必须给当前 TaskQueue active job 留下永久取消标记。只比较 chatKey 无法识别 A → B → A；宿主保存或世界书 I/O 的迟到结果必须由既有 task guard 拒绝。

固定文本普通块必须用对应结束标签闭合。缺失结束标签、嵌套新块、错配结束标签或孤立结束标签属于协议失败，不得把已读字段作为半成品提交。`<MA_REPLACEMENT>` 原文块只识别自身结束标签，正文内出现其他协议标签仍作为原文保留。

## 本版核心边界

rc.58 不改变对象准入、事实数量、变化幅度或旧快照修订原则。变化只发生在模型输入的旧数据表达：完整内部快照继续由插件持有，模型只接收全局对象短目录与少量高相关工作副本。

`<MA_DIRECTORY>` 只提供表格、规范名称和少量别名，用于身份识别；它不是白名单。目录未列出的正文新对象仍可输出，插件在完整内部快照中再次按名称和别名查重后决定沿用旧 ID 或创建新 ID。

`<MA_CONTEXT_ROW>` 只包含直接命中对象、当前时空、活跃事件、当前角色及直接关联对象的精简当前切面。对象未进入工作副本但本轮确实变化时，模型仍可按目录名称返回变化字段；插件从完整旧快照保留未返回字段。

rc.57 的单项表达收紧与确定性差异抑制继续生效：不存在总条数硬上限；剧烈变化必须完整返回；空白、末尾标点和数组换序不构成真实更新。

rc.52 将“模型生成结构化数据”和“插件管理内部数据”彻底分离；rc.53 在此基础上把“同表同名唯一”提升为对象层、历史层和世界书出口共同遵守的硬约束。

模型通信不再使用 JSON。模型输入中的相关事实、旧对象索引和旧行全文由插件序列化为 `<MA_CONTEXT_…>` 固定文本块；模型输出使用任务对应的固定标签文本。模型不再返回 JSON、JSON Schema 对象、稳定 ID、整表快照或数据库补丁。

JSON 只保留为插件内部的设置、快照、任务、表格、迁移和诊断存储格式，不再作为模型通信协议。



## 历史恢复检查点边界（rc.56）

历史核心重建采用逐消息提交。每条状态提取成功后，其 artifact、内部事实、快照和处理凭证立即固化；后续消息失败不得回滚或重新调用已经成功的前缀。

`historyRecovery.currentIndex` 是下一条待处理或本次失败消息，`completedCount` 是当前可处理消息序列中已经成功固化的数量。再次执行恢复时，必须按 `currentIndex` 在当前可处理序列中的位置切分后缀，不得重新从 `historyInvalidation.startIndex` 开始。

只有再次发生历史编辑、换页或删除并触发新的历史失效时，旧检查点才作废。若核心阶段已完成而派生总结或世界书失败，重试直接进入派生恢复。


## 运行时队列边界（rc.55）

模型空响应、协议错误和上游失败必须进入终态并释放任务去重键。持久化阶段记录只用于展示和恢复，不得单独充当运行时互斥锁；`queued/running` 是否仍然有效，以当前 `TaskQueue` 中同聊天、同消息、同任务类别的真实活动记录为准。

玩家显式提交小总结或大总结时，手动任务可以接管同聊天的陈旧自动派生链：尚未执行的自动总结和自动同步被取消，正在运行的自动总结被请求取消。已开始的其他高优先级核心任务和世界书物理写入不被强制并发打断。

## 模型职责

模型只负责读取固定文本上下文，并填写固定文本壳中的自然语言内容：

- 审核：`<MA_AUDIT>`、`<MA_VIOLATION>`、`<MA_REPLACEMENT>`；
- 状态提取：主协议为 `<MA_TURN>`、`<MA_CHANGE>`；旧 `<MA_FACT>`、`<MA_ROW>` 仅作兼容输入；
- 小总结与大总结：`<MA_SUMMARY>`；
- 定向修正：返回完整修正正文，不包数据库结构。

模型不得：

- 生成或选择稳定 ID；
- 直接指定内部行键、事实键或事件键；
- 返回完整 JavaScript/JSON 对象；
- 决定覆盖数据库中的具体记录；
- 复制整张旧表；
- 通过未注册字段扩展内部结构。

固定标签和字段名是协议边界，字段内容允许使用自然语言。缺少必要标签、重复唯一槽位、出现未知槽位或无法闭合的关键结构时，插件拒绝提交，不猜测写入。

## 插件职责

插件是唯一的数据管理员，按以下顺序处理模型输出：

1. 解析固定标签块；
2. 规范化字段名、空白、分隔符和列表；
3. 根据表格、对象名、关键词、别名、事实和事件关联匹配已有对象；
4. 合并同一次响应中的重复条目；
5. 对有充分证据的历史重复行执行收拢；
6. 沿用已有稳定 ID，或由插件为真正的新对象生成 ID；
7. 应用人工锁、基础层保护、字段注册表和历史事务校验；
8. 转换为内部对象并写入聊天快照；
9. 在事务成功后进入总结、世界书同步和诊断链路。

任何模型文本都不得绕过解析器直接写入内部表格。

## 对象身份与去重边界

对象身份不依赖模型提供的 ID，并遵守以下硬约束：

> 同一聊天、同一表格、同一规范名称只能存在一个对象。

空白、常见分隔符、括号、书名号和大小写差异由全链共用的名称规范化函数处理。状态解析、快照归一化、保护层合并、手工编辑、历史迁移和世界书发布不得各自定义不同的“同名”算法。

不同事实、事件、时间和状态属于同一对象的不同切面，应进入 `currentFacts`、`currentStates`、`recentHistory`、`solidifiedHistory`、事实 ID、事件 ID 与其他注册字段，不得通过创建同名行表达。

插件按以下层级维护唯一身份：

1. 同一次模型响应中的同对象、同字段 `<MA_CHANGE>` 先按最终明确结果归并；旧同名 `<MA_ROW>` 仍按兼容规则合并；
2. 同一快照中的同表同名行无条件归并；
3. 不同历史快照中的同一规范对象沿用首次稳定 ID；
4. 手工新增或改名命中同名对象时更新原对象，不新增第二行；
5. 关联对象引用和玩家焦点 ID 随旧 ID 映射改写；
6. 世界书待发布计划按完整托管名称执行最终唯一化；
7. 当前聊天已有同名托管副本确定性保留一个 UID，并删除其余副本。

人工基础保护和完全锁定仍决定合并后的内容优先级，但不赋予同名行共存权。真正需要区分的同名对象必须在首次建立时使用不同的规范名称，世界书不使用 ID 后缀临时掩盖身份冲突。

旧聊天通过一次性 `uniqueObjectNamesV31` 迁移扫描全部历史快照。迁移保存旧 ID 到规范 ID 的别名，即使聊天已保存而 metadata 保存失败，也能在刷新后继续修复焦点和引用，不会重新制造重复。

## 内部 JSON 边界

以下内容继续使用插件内部 JavaScript 对象和 JSON 序列化：

- 扩展设置与动态表格注册表；
- 聊天状态、事实账本和对象快照；
- 小总结、大总结和消费记录；
- 任务队列、请求轨迹和诊断导出；
- 世界书逻辑键、物理键和同步映射；
- 版本迁移与历史恢复检查点。

这些 JSON 全部由插件代码产生、校验和迁移。模型无法直接构造或提交这些对象。

## 已移除的旧模型协议设施

rc.52 运行时不再包含：

- OpenAI `response_format` / `json_schema` 请求；
- 通过 `JSON.stringify` 向模型传递内部事实、索引或旧行；
- strict JSON Schema 模型输出；
- Schema 能力缓存与无 Schema 回退；
- “兼容模式”请求开关；
- 模型 JSON 提取、代码块清理和 JSON 修复重试；
- 由模型生成稳定 ID 或完整表格补丁的路径。

旧设置键仅在加载旧配置时被删除，用于迁移清理，不会重新启用旧协议。

## 人工数据与基础层保护

固定文本协议不改变既有数据保护边界：

- 人工锁定行不得被自动流程删除或覆盖；
- 非空 `baseContent` 原则上不由普通状态提取改写；
- 明确重定义、毁灭、重建或人工编辑才可改变基础定义；
- 未返回的旧行继续保留，空响应不等于清空整表；
- 未注册字段不会进入内部对象；
- 焦点只影响世界书常驻状态，不改变对象身份、排序或裁剪规则。

## 历史重建事务边界

历史失效后，普通状态整理、总结与世界书同步继续受锁约束。

重建过程必须：

- 使用同一聊天范围快照；
- 每次提交前验证聊天、历史修订和消息指纹；
- 失败时保留最后已确认快照和恢复检查点；
- 不把未完成重建的数据发布到世界书；
- 只有重建、发布和必要回读全部成功后才清除历史锁。

固定文本解析失败属于当前消息处理失败，不允许退回旧 JSON 协议，也不允许将原始文本当成有效表格数据提交。

## 世界书同步边界

世界书仍是长期记忆发布层，不是模型输出的直接落点。

同步只消费已经通过插件解析、去重、校验并提交的内部状态。逻辑键、物理键、聊天归属、人工条目保护、保存后回读和重复归并规则继续生效。

## 请求与诊断边界

当前聊天连接和 Connection Profile 均发送普通文本请求，不附带 JSON Schema 或结构化输出字段。

诊断记录协议类型、连接 lane、任务类型、耗时、状态和错误分类，但不记录玩家正文、完整模型输出或 API 密钥。连接成功只证明对应实测请求成功，不以服务对象存在代替真实请求结果。

## 验收结论

rc.53 保留 rc.52 的固定文本模型协议，并完成对象唯一性闭环：模型负责内容理解，插件负责身份、切面合并、分发、稳定 ID、历史迁移和世界书唯一发布。相同规范名称不会再以多个表格行或多个当前聊天世界书条目存在。
