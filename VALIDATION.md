## rc.40 Schema 缓存回归

本版只验证一个已复现 Bug：Schema 兼容 fallback 在内容尚未解析前就提交拒绝缓存，导致后续状态重试永久直走 plain，并反复产生截断 JSON。

本地新增验证：

- Schema 400 后 fallback 被截断时，不提交具体 Schema 拒绝缓存；下一次任务重新发送 Schema；
- fallback 完整并通过目标 Schema 后，下一次才允许内存级 plain 绕过；
- 已缓存绕过的 plain 输出一旦截断，立即清除缓存，下一次重新发送 Schema；
- rc.35 遗留的持久化具体 Schema 拒绝缓存升级时自动清除；
- 具体 Schema 拒绝不再写入 sessionStorage；
- 审核与连接测试也只有在 fallback 结果有效后才提交缓存。

实机只需重试失败的历史恢复。首个 state 请求应重新显示 `requestPurpose: schema`、`hasJsonSchema: true`、`jsonSchemaName: MirrorAbyssStateOperationsV37`，不应继续直接 plain。

## rc.39 统一实机验收

本版在 rc.38 总表基础上继续收口请求调度与历史状态，不要求分段实机。统一演示重点：

- 刷新后旧 queued/running 状态被对账，不永久残留；
- 最终状态、总结与世界书仍按原单 active 事务顺序提交。

本地新增验证：

- 5xx/超时不触发普通 JSON 重复请求；

## rc.34 多事件线记忆漏斗

已验证：

- 一次状态结果可包含多条互不关联事件线；每条线保留独立 `event_id`。
- 同一对象同时受多条事件影响时，`eventIds / factIds` 本地归并，不发生后写覆盖。
- 同一事件传播到人物、地区、事件和全局条目时，各独立信息点全部保留。
- 世界书精确重复只按稳定条目身份与完全相同内容处理，不再按共享 `fact_id / event_id` 删除其他对象。
- 八类默认表均具有对象定义、现行事实、当前状态、近期经历和历史事实层。
- 状态提取不得改写近期经历和历史事实；模型无需输出 `factIds / eventId / eventIds / recall`。
- 小总结可在一次请求中批量处理最多 8 条到期事件线，并逐 `event_id` 校验完整返回。
- 大总结可在一次请求中批量处理最多 8 条成熟事件线，不同事件的小总结数量不得合并凑阈值。
- 小总结将内容传播到同一事件及其真实受影响对象的 `recentHistory`。
- 大总结删除已消费的近期文本，并将长期结果写入 `solidifiedHistory`。
- 已结束事件条目不会被沉降删除，仍作为独立事件信息点存在。
- 小总结和大总结不再发布独立世界书条目，不产生第二套关键词、常量或语义触发。
- 启用语义时，事件条目整体使用 vector 召回；人物、地区、物品等对象保持 trigger；玩家焦点角色保持 constant。
- 完全锁定条目不会被总结层新增、删除或迁移内容。
- 同批总结写入、内部凭证提交和失败回滚保持事务边界。

## 实机验收重点

1. 在同一条 AI 正文中同时写入两个无关事件，例如“北境屠龙”和“南港走私”。
2. 诊断中应只有一次 state 请求，并生成两个不同 `event_id`。
3. 人物、地区、事件、全局等受影响条目分别更新；屠龙条目不得出现走私内容，走私条目不得出现真龙内容。
4. 达到小总结条件后，两条事件可在一次 smallSummary 请求中处理，但结果逐事件分区写入各自条目近期经历。
5. 仅让其中一条事件结束，另一条仍 active；大总结只能固化成熟事件线。
6. 世界书中不应出现新的 `small:event:*` 或 `large:event:*` 正式条目。
7. 开启语义后事件条目应为 vector；对象条目仍按名称触发；焦点角色仍为 constant。
8. 导出诊断，核对 state 的 `promptChars / requestMs / responseTokens` 是否明显低于 rc.31–rc.33，并确认后续世界书同步能够实际完成。
