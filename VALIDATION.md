# Mirror Abyss 1.2.0-rc.17 验收记录

## 自动检查

- TypeScript 严格类型检查通过，并启用未使用变量与参数检查。
- esbuild 浏览器 ESM 构建通过，最终 `index.js` 通过 Node 语法检查。
- 任务优先级测试通过：活动任务完成后，关键任务先于排队中的大总结。
- 任务去重、排队取消、禁用/启用代次和空闲释放测试通过。
- 请求通道测试通过：同一通道按优先级串行，排队请求可取消，不同原生通道互不锁死。
- 触发边界测试通过：只接受 `is_user === false` 的非系统非空正文；玩家、系统、空白和来源不明消息均拒绝。
- 自适应小总结密度策略测试通过。
- 定向修正集成测试通过：`block` 审核仍调用修正与复审，复审通过后原位替换正文和活动 swipe，不新增消息。
- 候选失败隔离测试通过：未提交候选不会覆盖当前正文审核，也不会改动聊天正文。
- manifest、运行时版本和流水线版本一致。
- 运行包不包含独立 API 地址、密钥存储、供应商 `fetch` 客户端、localStorage/localForage 或生成拦截器。

## 静态调用链复核

- 正文关键链和小总结、大总结、世界书发布已拆开，不存在活动任务内部再次等待同一任务队列的嵌套死锁。
- 排队中的大总结优先级低于新正文关键链。
- 同一连接 Profile 仍为单请求通道；没有开放无界并发。
- 插件禁用、聊天切换、历史变化和重置会取消模型请求或排队任务，并通过提交守卫拒绝旧结果。
- 外部 `signal` 与插件控制器已联动，插件主动取消不会再被外部信号覆盖。
- 世界书只更新、暂停或清理同一 `chatKey` 的托管条目。
- 历史事实与状态重算成功后，即使总结或世界书失败，也保留核心结果并显示部分失败提示。
- 诊断数据不保存提示词、正文、模型响应全文或密钥。

## SillyTavern 人工验收项

1. 安装后确认版本显示为 `1.2.0-rc.17`，设置入口与工作区可打开。
2. 分别测试当前聊天连接和一个 Connection Profile。
3. 发送玩家消息，确认不会触发镜渊任务；生成一条 AI/角色正文，确认只触发一次关键链。
4. 在大总结排队时继续生成正文，确认新正文排在尚未开始的大总结之前。
5. 生成途中切换聊天，确认旧任务取消，且新聊天同索引消息没有被写入旧结果。
6. 编辑、删除或 swipe 旧消息，确认世界书暂停并出现重算入口。
7. 重算后确认事实与状态恢复；人为制造总结失败时，核心重算仍保留并给出警告。
8. 禁用再启用插件，确认没有重复事件监听、重复 UI 或旧排队任务复活。
9. 检查诊断报告，确认只有阶段和耗时信息，没有正文、提示词或密钥。

## rc.15 新增检查

- 最新 swipe 局部失效测试通过：切换最后一条 AI 正文后，只删除该正文旧 artifact、旧 processed key、直接相关小总结及依赖大总结；上一份成功快照继续作为基线。
- 历史边界测试通过：最后一条 AI 正文重新生成不会留下 `historyInvalidation`；旧消息 swipe 仍走原有历史重算链。
- 迟到事件竞态测试通过：人工定向修正已经进入模型请求后再收到同一最新正文的 `MESSAGE_SWIPED`，当前请求不会被取消，复审通过后仍原位替换。
- 升级恢复测试通过：rc.14 遗留的“最新 swipe 历史失效”会在人工重新整理前安全清除，后续总结和世界书不再被假失效阻断。
- 修正提交守卫测试随完整回归通过：正文指纹改变后，旧 artifact 在进入修正及每次修正尝试前都会被拒绝。

### 真实操作验收

1. 保持插件已安装并启用审核、状态和世界书。
2. 在最后一条 AI 正文点击 SillyTavern“重新生成/切换 swipe”。
3. 新正文出现后立即在镜渊点击“整理最新正文”。
4. 若审核阻断，应依次显示：审核阻断 → 修正处理中 → 修正成功 → 审核通过 → 状态处理中。
5. 确认没有出现“历史数据需要重算”，没有要求选择历史起点。
6. 确认修正后的文本替换当前活动 swipe，消息数量不变。
7. 确认状态、总结与世界书基于新正文恢复，旧正文事实不再存在。
8. 再对一条非最新 AI 消息切换 swipe，确认仍出现历史失效和重算入口，以证明边界没有放宽到旧历史。

## rc.16 新增检查

- 官方事件语义复核：SillyTavern 删除消息后发出 `MESSAGE_DELETED(chat.length)`，因此插件按删除后剩余 artifact 一致性判断局部删除或历史删除，不依赖事件参数猜测原位置。
- 末端删除局部失效测试通过：删除最后一条已整理 AI 正文后，不创建全局 `historyInvalidation`；旧 processed key、快照指针、直接小总结及依赖大总结被清除。
- 旧历史删除边界测试通过：删除较早正文后，后续 artifact 因索引/身份错位而进入原有历史重算链。
- 整理中删除竞态测试通过：正在运行的旧正文审核请求收到取消信号并以 `AbortError` 结束；重新生成的新正文随后可完成审核、状态提交，且不会继承旧历史失效。
- 升级恢复测试通过：rc.15 遗留的无起点 `deleted` 假失效在人工整理前被清除，旧 orphan key 和相关总结不会残留到新正文状态中。
- 完整 TypeScript、任务调度、请求取消、定向修正、swipe 边界、总结策略、构建与语法回归通过。

### 真实操作验收

1. 点击“整理最新正文”，在审核或修正仍运行时删除最后一条 AI 正文。
2. 确认旧请求结束，旧正文结果不会重新写回聊天，也不出现要求选择历史重算起点。
3. 重新生成新正文并点击“整理最新正文”。
4. 确认新正文正常经历审核、必要的定向修正、状态、总结和世界书发布。
5. 检查旧正文的事实、processed key 和相关总结已经消失。
6. 再删除一条后面仍有已整理正文的旧历史消息，确认仍出现历史失效和重算入口。

### 玩家运行逻辑验收

1. 关闭镜渊工作区，生成正文，确认后台审核和状态仍完成。
2. 打开工作区并切换总览、状态表、总结、同步、诊断标签，确认模型请求数不增加。
3. 生成违规正文，确认消息面板收到“审核阻断→修正处理中→修正成功”，后台完成复审和状态。
4. 关闭工作区后继续生成正文，确认后台继续处理且 UI 订阅通知停止。
5. 在工作区关闭时手工状态服务仍可保存人工锁定行并触发世界书同步。
6. 重新打开工作区，确认直接读取最新 artifact、总结和同步状态，不重新调用模型。
7. 从简易设置和工作区分别关闭/启用插件，确认行为一致，不出现一处关闭一处仍运行。
