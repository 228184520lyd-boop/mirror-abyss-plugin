# Mirror Abyss rc.39 请求调度与历史状态边界报告

版本：`1.2.0-rc.39`  
流水线：`ma-pipeline-41`

## 一、实机证据

rc.38 诊断证明，连接测试变慢并不是模型本身突然变慢，而是插件调度等待：

- 状态结构化请求实际执行约 60 秒并返回 HTTP 504；
- 连接测试本身只执行约 3.4 秒，但 `transportWaitMs` 达到约 42.6 秒；
- 连接测试与状态提取共用 `current-chat` 串行通道，因此只读诊断被长业务请求阻塞；
- 更严重的是，连接测试会插入状态 Schema 失败与无 Schema 回退之间，打断同一业务链的连续性；
- 一次历史恢复状态任务曾先等待 Schema 504，再发送无 Schema请求并收到 502，总耗时约 121 秒。

诊断还显示，历史恢复进度被写入 `lastSyncStatus / lastSyncError`，导致“最近世界书同步”错误显示为失败，而不是保留上一次真实同步结果。

## 二、根因

### 1. 调度通道职责混合

旧实现按物理连接建立单一串行通道，没有区分：

- 正式业务请求：审核、修正、状态、总结；
- 只读诊断请求：连接测试、能力探针。

二者既不共享业务事务，也不应互相阻塞，却被放在同一队列。

### 2. Schema 回退触发条件过宽

Schema 请求遇到 500、502、504、超时等上游故障时，旧实现仍可能继续发送无 Schema请求。此类错误与 Schema 兼容性无关，重复请求只会增加等待和上游负载。

### 3. 取消边界缺少请求元数据

拆分诊断通道后，旧的全局活动请求取消机制会误伤并行连接测试。活动请求必须携带任务和请求类别，正文抢占、历史失效、聊天切换分别按不同范围取消。

### 4. 历史控制流污染同步结果

历史失效和恢复是发布门禁，不是一次世界书写入失败。将恢复进度写入 `lastSyncError` 会掩盖上一次真实保存结果，也使诊断无法区分“暂停发布”与“保存失败”。

## 三、rc.39 修复

### 1. 业务通道与诊断通道分离

每个物理连接最多同时存在：

- 一条串行 `business` 通道；
- 一条串行 `diagnostic` 通道。

同类请求仍严格串行；只读诊断可与一个业务请求并行。正式状态、总结和世界书写入顺序不变，不引入并发提交。

诊断新增：

- `connectionLane`
- `requestClass`
- `requestOrigin`

典型通道：

```text
current-chat:business
current-chat:diagnostic
profile:<id>:business
profile:<id>:diagnostic
```

### 2. 上游故障不再触发无 Schema重复请求

只有以下情况允许一次普通 JSON 兼容回退：

- 接口明确不支持结构化输出；
- 当前 Schema 被 400 类兼容错误拒绝；
- Schema 返回空对象。

以下情况直接失败，不发送第二次业务请求：

- HTTP 5xx；
- 超时；
- 限流；
- 鉴权失败；
- 主动取消；
- 其他与 Schema 兼容性无关的传输故障。

因此持续 504 时，单次任务最多消耗一个网关周期，不再从约 60 秒扩大到约 120 秒。

### 3. 请求取消改为定向规则

活动请求现在登记 `task + requestClass`：

- 新正文只取消自动 `smallSummary / largeSummary` 业务请求；
- 历史失效取消当前聊天的业务请求，不取消只读诊断；
- 聊天切换、插件禁用和重置仍可取消全部请求；
- `Cancelled by stop event` 等宿主取消信息统一分类为 `cancelled`。

### 4. 历史恢复与真实同步状态分离

- `historyInvalidation / historyRecovery` 表示历史门禁和恢复阶段；
- `lastSyncStatus / lastSyncError / lastSyncAt` 只表示真实世界书写入结果；
- 恢复期间同步诊断显示“暂停”，并保留上一次真实同步时间；
- rc.38 及更早版本写入的控制流错误字符串会自动迁移；
- 暂停旧世界书条目失败时持久记录 `historyInvalidation.pauseError`，不会只显示一次 toast；
- 诊断导出会脱敏历史与同步错误。

## 四、本地验证

rc.39 专项回归覆盖：

- 长业务请求运行时，诊断连接测试不等待 business 通道；
- 同一类别内部仍串行；
- 业务 Schema 504 后不创建 fallback trace；
- 诊断请求不会插入业务 Schema→fallback 兼容链；
- 自动总结抢占不取消诊断；
- 历史失效取消业务请求但保留诊断；
- 宿主 stop event 分类为 cancelled；
- 历史恢复期间同步显示暂停，上次实际同步结果不被覆盖；
- 旧污染状态自动迁移；
- 嵌套错误字段在诊断中脱敏。

完整自动验证输出见 `RC39_FULL_VERIFY_OUTPUT.txt`。

## 五、仍需实机确认的边界

本地可以证明插件队列等待被移除，不能保证中转站允许真正并发请求。实机诊断应分别观察：

- `transportWaitMs`：插件内部等待；
- `requestMs`：上游实际执行时间。

预期连接测试的 `transportWaitMs` 接近 0；若 `requestMs` 仍很长，则瓶颈位于中转站或供应商，而不是镜渊调度。
