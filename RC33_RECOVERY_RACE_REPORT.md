# Mirror Abyss 1.2.0-rc.33｜历史恢复自动任务竞态修复

## 实机证据

rc.32 实机诊断显示：

- 历史恢复派生任务于 `18:14:32.478` 完成；
- 相同正文的普通 state 任务已于 `18:14:32.065` 入队；
- 该任务在恢复完成、历史锁解除的同一时刻开始运行；
- 随后再次发送约 6,497 字符提示、2,400 token 的 state 请求。

这说明宿主保存或消息生命周期事件在恢复期间产生了陈旧自动任务。旧任务入队时被活动恢复任务挡在队列中，恢复结束后才获得执行机会，因此仅依赖历史锁不能阻止它。

## 修复范围

本版不修改模型提示、状态 JSON 协议、总结规则、世界书条目结构或历史恢复业务结果，只修任务调度与幂等边界。

### 1. 恢复期间抑制自动调度

`scheduleMessage()` 在定时器真正触发时重新读取聊天状态。存在 `historyRecovery` 时直接退出，不创建普通处理任务。

历史重建开始时清除当前聊天已有的自动定时器，防止恢复前创建、恢复中才触发的延迟任务。

### 2. 恢复提交前清理陈旧任务

历史核心、总结和世界书全部提交成功后，恢复任务在退出队列前：

- 再次清除当前聊天的自动定时器；
- 取消恢复期间排入、且指向本次已恢复消息键的自动任务；
- 手动任务和历史恢复任务不在该清理范围内。

清理发生在活动恢复任务内部，因此队列尚未开始下一个 pending 任务，不存在“先启动、后取消”的空窗。

### 3. 启动时执行提交幂等校验

即使宿主事件迟到到历史锁解除之后，自动任务启动时仍检查：

- `chatKey`；
- 消息稳定键；
- 当前正文指纹；
- 消息 artifact 是否已有成功 snapshot；
- `processedMessageKeys` 是否已包含该消息键。

全部一致时，本次任务进入 `skipped`，不调用审核模型、状态模型，不再次提交事实或世界书。

手动显式整理不受这条自动幂等规则影响。

### 4. 新增 skipped 任务语义

新增 `TaskSkippedError`：

- `blocked`：业务前置条件未满足，例如历史锁要求先重建；
- `cancelled`：排队任务在启动前已失效；
- `skipped`：输入已经由当前流水线正式完成，无需重复执行；
- `failed`：任务真实执行失败。

### 5. 增强诊断

任务诊断新增：

- `triggerSource`；
- `messageKey`；
- `messageFingerprint`；
- `historyRevisionAtEnqueue`；
- `historyRecoveryPhaseAtEnqueue`；
- `automatic`；
- `skipReason`。

事件来源会区分 `message-received`、`history-edited`、`history-swiped`、派生总结与派生同步。

## 新增回归测试

`tests/rc33-recovery-auto-race.ts` 按实机顺序模拟：

1. 建立正常状态与世界书；
2. 修改历史并启动恢复；
3. 暂停世界书保存，使恢复任务继续占用队列；
4. 插入一条由 `message-received` 触发的相同正文自动任务；
5. 释放世界书保存；
6. 验证陈旧任务在启动前取消，模型调用数不增加；
7. 在锁解除后再次模拟迟到事件；
8. 验证任务进入 `skipped`，模型调用数仍不增加。
