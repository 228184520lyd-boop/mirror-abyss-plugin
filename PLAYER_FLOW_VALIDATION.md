# Mirror Abyss 1.2.0-rc.17 玩家运行逻辑与 UI/业务分离验证

## 结论

rc.17 达到“运行分离、克制分层”：UI 是否打开不会决定后台是否运行；打开、关闭和切换只读标签不会触发模型；UI 的主动命令仍明确调用业务服务。没有引入前端状态框架、独立后端或额外数据库。

## 本次迭代

1. 审核与定向修正只发布展示状态，不再查询或修改消息 DOM。
2. 消息面板订阅展示状态，负责正文隐藏与违规标记。
3. 简易设置和工作区统一通过 `setPluginEnabled()` 控制插件运行。
4. 手工状态行统一通过 `saveManualStateRow()` / `removeManualStateRow()` 提交。
5. 核心业务目录不导入 UI；`bootstrap/app.ts` 作为组合入口保留 UI 依赖。

## 玩家逻辑模拟

模拟顺序：

1. UI 关闭，玩家生成第一轮正文。
2. 后台完成审核与状态提取。
3. UI 打开并读取总览、状态、任务与聊天状态。
4. 玩家生成包含违规内容的第二轮正文。
5. 审核阻断，发布违规标记；定向修正；候选复审；原位替换；状态提取。
6. 玩家主动执行小总结、大总结和世界书发布。
7. UI 关闭，玩家生成第三轮正文。
8. 后台继续完成审核和状态。
9. UI 关闭时保存一条人工锁定物品状态，并同步世界书。
10. 通过统一服务关闭插件，确认不新增模型调用；重新启用。
11. UI 再次打开，只读取最终 artifact、总结和同步状态。

- 最终托管世界书条目：6。
- 最终有效 processed key：3。

## 仍需真实 SillyTavern 验收

本模拟证明插件内部的调用、订阅、存储与队列关系。以下内容仍只能在真实环境确认：

- 手机浏览器上的视觉刷新与触控。
- SillyTavern 实际 DOM 结构是否匹配消息选择器。
- 不同 SillyTavern 版本的事件到达顺序。
- 真实模型供应商的取消、超时和响应延迟。
- 世界书编辑器即时刷新。

## 判断

本次迭代解决的是维护层耦合，不是增加功能。UI 已不再直接拼装任务队列、请求取消和状态保存步骤；后台也不再直接控制审核 DOM。当前无需继续引入更复杂的 UI 架构。
