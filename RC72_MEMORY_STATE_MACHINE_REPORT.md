# RC72 记忆状态机集中区审计与收拢报告

## 审计范围

本轮采用“全项目浅筛、状态机集中区深筛”。深筛链路为：

```text
状态事实输入
→ 对象身份与唯一化
→ 快照字段更新
→ 特殊表规则
→ 条目生命周期
→ 小总结／大总结覆盖
→ 物理删除
→ 世界书同步
```

没有扩展原子事实递归，没有让模型参与重要性、未来价值、合并或删除判断，也没有将完整仓库正文发送给模型。

## 集中区已修复

### 1. 快照模块职责过载

原 `snapshot.ts` 同时承担稳定 ID、引用改写、时空单例、生命周期、覆盖审计与旧墓碑清理。现拆分为：

- `object-identity.ts`
- `special-table-rules.ts`
- `entry-lifecycle.ts`
- `memory-state-machine.ts`

`snapshot.ts` 保留通用快照与兼容层，避免外围模块一次性重写。

### 2. 总结覆盖与删除顺序

新增 `finalizeSummarySettlement`。调用前检查事务副本中的事实是否已写入 `consumedBySmallSummaryId` 或 `solidifiedByLargeSummaryId`。缺少标记直接报错，不能先删除再补状态。

### 3. 临时 ID 与稳定 ID 脱节

新增 `alignPatchRowsToCanonicalSnapshot`。状态模型产生的临时对象 ID 会按既有 ID 映射、唯一标题、事件或事实引用绑定到规范稳定对象，使恢复、待结算和同轮更新命中同一条目。

### 4. 旧墓碑误删风险

旧 `absorbed / retired` 归一为 `settling` 后，仅当条目存在至少一条可核验事实，且全部事实已经被小总结或大总结覆盖、没有活跃事实、宿主分发完整且不受保护时才删除。无事实凭证不删除。

### 5. 无操作迁移产生副作用

没有旧生命周期数据时，旧墓碑清理直接返回原快照，不再顺带去重、重排或制造新的保存差异。

### 6. 生命周期读取口径分裂

世界书发布、事件画像和关系图谱改为使用统一参与状态辅助函数。生产写入只使用 `active / settling`；旧状态判断集中在迁移模块。

## 全项目浅筛

### 已随集中区修复

- 事件画像曾直接读取具体生命周期字符串，已改用统一参与状态；
- 当前时空规则从通用快照流程拆出，并增加幂等测试；
- 当前 schema 快速路径补齐 `memoryStateVersion` 和 `entryLifecycleV33`，避免每次读取重复迁移；
- 状态补丁稳定 ID 对齐避免同轮产生重复对象或生命周期漏触发。

### 发现但未扩大处理

- 旧固定文本解析器仍兼容历史生命周期字段，这是旧响应兼容入口；生产流水线继续忽略模型生命周期意见，本轮不删除兼容解析。
- 场景兜底仍沿用现有规则。本轮没有证据表明它导致当前集中区故障，因此未扩展为场景体系重写。
- 状态请求预算和 504 分段沿用 1.3.7 已验证机制，本轮不重新设计协议。

## 结论

问题确实集中在状态机阶段契约，而非单一模型提示词或记忆网络 UI。本轮将输入、输出、调用顺序和允许转移收敛为确定协议；外围功能通过兼容导出保持原行为。
