# rc.38 固定问题台账总审计与统一实机计划

版本：`1.2.0-rc.38`  
流水线：`ma-pipeline-40`

## 结论

本轮没有继续改动已经由 rc.37 实机验证成功的严格状态 Schema。工作范围回到最初 48 项问题台账，逐项核对真实代码、回归测试和可观测诊断。

本地能够确定并安全修复的问题已全部收口；必须依赖真实 SillyTavern、当前中转站、模型响应和世界书接口的项目，合并到一次统一实机演示；上游约 60 秒网关和 SillyTavern 不提供数据库事务等客观限制，明确保留为外部边界，不伪称已根治。

## rc.38 新增修复

1. **启动与监听器生命周期**
   - 启动重试只允许一个定时器；禁用、重启、成功取得上下文后均清理。
   - 事件卸载兼容 `off` 与 `removeListener`。
   - pipeline 和 UI 各自的合法监听器可以共存，但重复激活不得叠加旧实例。

2. **运行时中断状态对账**
   - 页面刷新、插件重启或切回聊天时，旧 artifact 中不可能继续运行的 `queued/running` 阶段转为 `cancelled`。
   - 被中断的历史恢复转为 `failed`，历史锁保留，用户可从失败位置继续。
   - 已成功的审核、状态、同步凭证不被撤销。

3. **真实任务终态**
   - 聊天切换、插件关闭、消息消失等不再通过 `return null` 被任务队列记成 success。
   - `CommitRejectedError` 归类为 cancelled。
   - audit、revision、state、summary、sync 的取消状态统一落盘。
   - 历史恢复没有产生总结时，summary 记为 skipped，不再伪装为新总结成功。

4. **无效大总结空任务**
   - 小总结完成后重新读取最新 ChatState，再判断是否真正达到大总结条件。
   - 只有一个小总结且阈值为四时，不再创建 0 次模型调用、几百毫秒 success 的大总结任务。

5. **高优先级正文不再长期等待自动总结**
   - 仍保持全插件单 active 业务任务，不引入并发写入。
   - 新正文到达时，仅允许取消同聊天正在运行的自动小/大总结，并取消其模型请求。
   - 已提交事实不回滚，未完成总结以后按当前事实重新归并。
   - state、sync、人工总结和历史恢复不会被该机制抢占。

6. **响应信封与空响应分类**
   - 补充 OpenAI tool call、Responses API output、Gemini parts/functionCall、对象 content 等返回位置。
   - `error/refusal/incomplete/finish_reason/blockReason` 不再被误送入业务 JSON 解析。
   - `No message generated` 作为独立上游/空响应错误。

7. **请求诊断分类**
   - 诊断新增 `errorKind` 与 `httpStatus`。
   - 区分 cancelled、timeout、empty、auth、rate_limit、upstream、request、unknown。
   - 任务诊断新增 `cancelRequestedAt/cancelReason`，内容继续脱敏。

8. **旧设置安全迁移**
   - 遗留 `auditFailAction: delete/withdraw` 统一迁移为 `hide`。
   - 次数、阈值、超时和召回参数执行边界归一化。

9. **打包类型检查**
   - 新增正式 `npm run typecheck`，并纳入 `npm run verify`。
   - 修正审核、修正、诊断和动态表格的过期 TypeScript 类型声明。

## 统一实机演示

不再逐补丁测试。安装 rc.38 后，在一个测试聊天和一个空白聊天中连续完成以下流程，再一次性导出诊断与录屏。

### A. 启动和基础链

1. 安装 rc.38，刷新页面；确认诊断版本为 `1.2.0-rc.38`。
2. 禁用再启用插件一次。
3. 生成一条简单角色正文。
4. 预期：只有一条 state 任务；audit 和 state Schema 直接成功；没有 state fallback；世界书 sync success。

### B. 自动总结让位与真实终态

1. 临时把小总结阈值设为 1，大总结阈值设为 4。
2. 生成一条会形成事件事实的正文，等待 smallSummary 进入 running 后立即生成下一轮正文。
3. 预期：旧自动 smallSummary 显示 cancelled 和 cancelReason；新 state 不需要等待旧总结完整返回；后续事实重新进入可总结集合。
4. 只形成一个小总结时，任务列表不得出现几百毫秒、零请求的 largeSummary success。

### C. 多事件与世界书

1. 一条正文同时发生两条无关事件，例如北境屠龙与南港走私。
2. 预期：一次 state 请求、两个 event_id；人物/地区/事件条目分别更新且不混线。
3. 连续同步两次，世界书 UID 数量不增加；人工条目保持不变。
4. 设置焦点甲、切换到乙、取消焦点；每次只有对应条目的 constant 改变。

### D. 历史恢复与入口阻断

1. 编辑一条较早角色正文，启动历史重算。
2. 恢复执行期间点击“整理最新正文”。
3. 预期：手动入口 blocked，不产生新的 state 请求；恢复在世界书回读成功后解除历史锁。
4. 恢复期间刷新一次页面，返回后应显示“历史恢复被中断/可继续”，不能永久显示 running，也不能显示数据为 0。

### E. 聊天切换竞态

1. 在聊天 A 启动 state 请求，模型返回前切换到聊天 B。
2. 预期：A 任务 cancelled；B 不出现 A 的表格、事实或世界书目标。
3. 返回 A，旧 running 状态已对账为 cancelled，可重新整理；重新整理成功后只提交一次。

### 最终提交材料

一次性提供：

- 完整录屏；
- rc.38 诊断 JSON；
- 世界书中镜渊托管条目的 UID、comment、constant 截图；
- 若出现失败，只需保留 `task/request state、triggerSource、errorKind、httpStatus、requestPurpose、hasJsonSchema、耗时`，不提供正文或密钥。
