# Mirror Abyss 1.3.9

发布日期：2026-07-21  
流水线：`ma-pipeline-72`

## 本版目标

本版不增加新的记忆能力，而是收拢已经存在的状态写入链。此前对象唯一化、时空单例、待结算、总结覆盖和世界书发布分散在多个入口，容易出现上一阶段输出尚未提交、下一阶段已经开始审计的情况。

1.3.9 将这条链固定为：

```text
明确事实
→ 稳定对象
→ 当前快照
→ active / settling
→ 总结覆盖
→ 物理删除
→ 世界书同步
```

模型职责不变：只提取正文已经明确的事实，不判断条目应否合并、退出或删除。

## 状态机规则

生产状态只有 `active` 和 `settling`。

- `active`：条目正常参与表格、关键词、向量、常驻或受控递归；
- `settling`：条目仍在同一快照中等待事实承接，但世界书参与权全部暂停；
- 物理删除：覆盖审计通过后删除容器，不保存 `deleted` 记录或永久墓碑；
- 恢复：只能在物理删除前由 `settling` 回到 `active`，并沿用原稳定 ID。

`absorbed / retired` 仅作为 1.3.4 等旧存档的迁移输入。

## 模块收拢

- `src/domain/object-identity.ts`：稳定 ID、唯一对象、本轮补丁 ID 对齐和引用改写；
- `src/domain/special-table-rules.ts`：每个聊天唯一 `spacetime_current`；
- `src/domain/entry-lifecycle.ts`：事实分发、参与状态、覆盖审计、恢复和物理删除；
- `src/domain/memory-state-machine.ts`：统一阶段顺序；
- `src/domain/snapshot.ts`：通用快照合并和兼容导出；
- `src/pipeline/summary.ts`：先提交总结覆盖标记，再调用状态机结算；
- `src/storage/repository.ts`：幂等、保守的旧存档迁移。

## 修复内容

1. 修复小总结或大总结已生成，但覆盖标记与结算调用顺序不一致的问题；现在缺少覆盖标记会直接拒绝结算。
2. 修复模型补丁携带临时 ID、快照已经继承稳定 ID后，恢复和生命周期动作无法命中同一对象的问题。
3. 旧墓碑没有事实引用时不再被迁移代码猜测删除；至少需要一条可核验且已被总结覆盖的事实。
4. 无旧墓碑时迁移函数保持原快照不变，不顺带归一化、重排或触发无关保存。
5. 世界书、事件画像和图谱不再分别解释生命周期字段，统一读取参与状态。
6. 时空单例成为独立特殊表规则，并保持多次执行结果一致。

## 兼容性

- 不改变 `<MA_TURN> / <MA_CHANGE>` 事实书记协议；
- 不扩大递归；
- 不进行模型全仓扫描；
- 不改变事件画像与关系图谱融合；
- 保留旧生命周期 API 的兼容导出；
- 旧存档升级时自动归一化生命周期和时空引用，保护人工、锁定和焦点条目。
