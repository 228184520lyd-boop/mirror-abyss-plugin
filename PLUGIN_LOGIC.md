# 镜渊插件概念与业务逻辑

适用版本：Mirror Abyss `1.3.9` / `ma-pipeline-72`

## 1. 定位

镜渊不是正文生成模型，也不是第二个角色扮演 Agent。它位于 SillyTavern 正文与世界书之间，承担审核、事实转换、对象维护、分层总结、历史恢复和记忆发布。

```text
主模型：创作正文、扮演角色、推进剧情
书记模型：从长文本中忠实提取明确事实
镜渊插件：确定性地匹配、整理、结算、压缩和发布
```

书记模型没有管理权限，也不应形成剧情观点。

## 2. 记忆漏斗

```text
正文
→ 本轮明确事实
→ 当前对象快照
→ 事件小总结
→ 阶段大总结
→ ST 世界书发布
```

各层职责不同：

- 当前快照回答对象此刻是什么状态；
- 内部事实记录哪些明确事件造成变化；
- 小总结承接近期事件线；
- 大总结固化跨阶段仍成立的历史结果；
- 事件画像提供派生浏览索引；
- 世界书是面向主模型的召回出口。

条目只是事实的当前工作容器。容器可以在事实被分发和总结承接后真实删除。

## 3. 聊天隔离与事务

每个聊天独立拥有快照、事实、小总结、大总结、世界书映射、焦点、历史检查点和任务状态。

所有异步任务绑定 chatKey、消息身份、正文指纹和运行代次。聊天切换、正文编辑或任务过期后，旧任务不能提交到新现场。状态提取成功后先提交核心快照和事实；后续总结或世界书失败不回滚已经确认的核心状态。

## 4. 书记模型边界

状态和总结模型只允许做两件事：

1. 从材料中提取已经明确发生、结束、替换或被证实的事实；
2. 将这些事实压缩为固定文本协议。

禁止模型：

- 评价事实是否重要或有未来价值；
- 解释人物动机、识别伏笔或补全背景；
- 枚举未知和“未解决问题”；
- 判断事件应否结算；
- 提出条目合并、退出、恢复或删除；
- 生成稳定 ID、事实 ID 或事件 ID。

即使旧模型仍返回 `entry_action` 或 `merge_*`，生产状态流水线也忽略这些意见。

## 5. 固定文本与确定性状态机

模型主协议返回 `<MA_TURN>` 和若干 `<MA_CHANGE>` 固定标签文本。每个变化只表达一个正文明确建立的对象字段结果；旧 `<MA_FACT>`、`<MA_ROW>` 继续作为兼容输入。插件负责：

```text
解析标签
→ 校验字段
→ 匹配已有对象
→ 合并同轮重复项
→ 分配或继承稳定 ID
→ 保护未返回字段
→ 写入快照和事实
→ 依据固定规则更新参与状态
```

固定协议只在诊断页只读展示。普通规则页开放的是信息提取边界和表头语义，而不是数据库控制权。

1.3.9 将事实落库后的中间层收拢为单向状态机：

```text
已确认事实与对象补丁
→ 对象身份与稳定 ID 对齐
→ 人工／锁定／基础层保护
→ 时空等特殊表规则
→ active / settling 生命周期迁移
→ 旧存档兼容迁移
→ 保存当前快照
→ 小总结／大总结先写覆盖标记
→ 覆盖审计与物理删除
→ 世界书同步
```

正式状态只有：

```text
active → settling → 物理删除
settling → active（结算前恢复）
```

`absorbed / retired` 只允许作为旧版本反序列化输入，不能由新流程继续写入。物理删除不是一个长期状态，不保存永久墓碑。

模块职责固定为：

- `object-identity.ts`：规范名称、稳定 ID 继承、本轮临时 ID 对齐和精确引用改写；
- `special-table-rules.ts`：当前时空单例等少量特殊表规则；
- `entry-lifecycle.ts`：参与状态、事实分发、覆盖审计、恢复与物理删除；
- `memory-state-machine.ts`：只负责编排阶段顺序和阶段契约；
- `snapshot.ts`：保留通用快照归一化、保护层和对象视图兼容函数；
- `pipeline/summary.ts`：在事务副本中先写总结层和事实覆盖标记，再调用结算入口；
- `lorebook-publish.ts`：只消费统一参与状态，不自行推断生命周期。

任何阶段不得绕过前置条件：`active` 不能直接删除；无事实凭证不能删除；总结覆盖标记尚未写入时不能审计；宿主未收到分发结果时不能删除源条目。

## 6. 状态请求预算与慢网关回退

1.3.7 将状态请求拆成三个白盒运行预算：上下文字符上限、最大输出 token 和 504 分段字符数。普通请求优先使用精简目录、少量相关工作副本和相关事实。

返回内容先在本地解析。代码围栏、空白、常见分隔符和必要字段完整的末块缺闭合可以安全规范化；纯结构错误最多进行一次紧凑修复。注册表越权、未知字段、禁用表格、对象歧义和生命周期指令属于语义错误，不得交给修复模型猜测。

遇到 502/503/504、超时或结构修复失败时，只拆分本轮正文并改用最小事实提示逐段提取。所有片段成功后统一生成事实和对象补丁并提交；任一片段失败都不产生半轮状态。

## 7. 增量处理，不进行模型全仓扫描

正常游玩只处理：

- 本轮正文直接触及的对象；
- 与本轮明确 event_id 相关的对象；
- 已经绑定该事件、正在待结算的对象。

状态模型只接收相关候选的短目录和少量高相关工作副本，不接收完整仓库正文。插件可以在本地遍历轻量索引筛选候选，但该过程不产生模型调用，也不会把全仓内容推向上游。

版本迁移、历史修复和用户主动一致性检查可以使用专用维护流程；它们不进入每轮游玩主链。

## 8. 对象身份与准入

默认对象包括时空、场景、角色、物品、事件、地点、全局变化、基础设定和自定义对象。

同一聊天、同一表格、同一规范名称只能有一个对象。匹配顺序为：

```text
同表规范名称
→ 已登记别名和关键词
→ 历史稳定名称映射
→ 无匹配且满足表格准入时创建
```

模型不决定 ID，也不因为本轮未提到对象而清空旧字段。

准入由白盒表格用途与提取规则约束。姓名、服装、站位、一次情绪、短动作或一句台词本身不足以把背景对象升级成长期角色。物品、场景等只要存在本轮连续性所需的明确状态，也可以成为临时工作条目；建档不等于永久保存。

## 9. 角色属性分层

1.3.6 起在现有角色表中建立最小结构化分工：

- **身份与对象定义**：对象是谁、是什么。普通外观、称呼、情绪和状态不得改写；只有正文明确纠正、重定义、转化、毁灭、重建或人工编辑时变化。
- **长期与现行事实**：正文明确成立、持续时间较长但仍可能被后续事实替换的属性与事实。不得从一次行为推断长期人格。
- **当前状态**：正在发生、短期或阶段性的状态，新事实可以更新、关闭或替换。
- **外观与表现**：服装、伪装、称呼、外观和他人明确认知，只表达表象，不反向修改身份。

关系状态和能力状态继续作为角色专用字段。该设计不是原子事实递归图，而是让现有表格字段各自承担明确职责。

## 10. 内部事实与当前快照

快照回答“对象现在是什么状态”，内部事实回答“正文明确发生了什么变化”。

每条成功正文先复制上一轮完整快照，再只应用本轮实质变化。未返回对象和字段保持不变；重复描述和措辞变化不产生新修订。

内部事实保留来源消息、事件关联、主体、发生结果、状态和总结消费关系。新版本不要求模型生成“未解决事项”；旧字段仅为历史兼容存在。

## 11. 插件侧增量结算

模型不再提出生命周期动作。插件根据本轮明确事实与固定规则机械派生：

- 已明确结束的事件可以进入待结算；
- 已明确结束或离开的场景可以并入事件或退出；
- 已出售、交付、消耗、损毁、丢弃等普通可替代物品，可以并入同事件或退出；
- 唯一、神器、证物、任务、关键、不可替代等物品不会因普通转移机械删除；
- 角色自动退出保持保守，不根据“以后似乎没用”判断；
- 人工、锁定、基础保护和焦点对象不参与自动退出。

待结算对象在 ST 世界书中停用，不参与常驻、关键词、向量或递归。小总结或大总结覆盖相关事实、宿主已承接持续结果且保护检查通过后，插件真实删除原表格容器和托管世界书条目。结算前对象再次被明确使用时可以恢复。

## 12. 小总结与大总结

小总结按事件线重写当前唯一有效的近期版本，只压缩材料中明确成立的事实。大总结读取已经整理好的事件总结，固化跨阶段仍成立的明确结果。

模型不负责判断“价值”，信息资格由白盒提示词中的具体事实类别规定；插件负责事件分组、版本继承、事实消费、硬上限校验和事务提交。

正常情况下，大总结不扫描混乱的完整仓库，而是处理每轮已经整理、结算和归类好的材料。

## 13. 记忆网络

“记忆网络”从当前快照、内部事实、小总结、大总结和待结算状态实时派生，包含事件画像与对象图谱。事件画像可显示：

- 事件状态；
- 事实数量和消息跨度；
- 关联实体与对象；
- 小总结版本和大总结状态；
- 待结算对象数量；
- 当前明确结果和内容规模。

对象图谱显示人物关系、物品、地点、场景、全局状态与事件之间的明确关联。事件画像产生的 event_id 和关联对象可派生为只读事件节点及连线。两种视图不保存第二份正文、不调用模型、不反向创建事实，也不决定事件结算。

## 14. 白盒硬字数上限

每种对象表、小总结和大总结都有可编辑硬上限。

- 对象世界书条目超限时，插件按字段优先级保留完整语义行；身份、当前事实和当前状态优先，低优先历史层先退出发布包。
- 小总结或大总结超限时拒绝提交，进入可重试失败；不把超长文本直接尾部截断后伪装成成功。
- 内部事实账本不会因为世界书展示预算被无规则删减。

上限是发布与总结边界，不是让模型凑字数的目标。

## 15. 世界书与召回

世界书是向主模型发布长期记忆的出口。插件按对象生成托管条目，并执行稳定条目复用、名称唯一化、当前聊天范围更新、召回模式设置、总容量裁剪、保存回读和界面刷新。

当前常驻、关键词、向量与递归结构保持不变。递归只用于受控地带出相关事件或总结，不扩展为属性和原子事实的全图遍历。焦点只控制对应世界书条目的 constant 属性。

## 16. 历史恢复与队列

历史编辑、删除或 Swipe 会使后续快照失效。镜渊从最早失效位置裁剪受影响事实和总结，暂停世界书发布，并按消息顺序重建。每条成功后保存检查点，中间失败后从失败消息继续。

历史流程由 `workflow/history-workflow.ts` 统一解释和推进：失效、重建核心、恢复总结、发布世界书、失败、部分完成与解除锁定只有一个流程入口。底层暂时仍保存两个兼容字段，但外围模块不得直接形成第二套转移规则。

审核、修正、状态和总结进入 TaskQueue；模型请求按物理连接进入请求 lane。持久化工作流表示业务进度，TaskQueue 只表示当前运行任务，二者不能互相替代。

## 17. 核心原则

```text
模型只说正文已经明确的事实
插件只按确定规则管理事实

快照负责当前
事实负责依据
小总结负责近期连续性
大总结负责长期固化
事件画像负责浏览
世界书负责发布与召回

每轮增量整理
事实先分发再删除容器
身份与表象分层
上限白盒可调
递归保持克制
```
