# 镜渊插件概念与业务逻辑

## 1. 定位

镜渊不是正文生成模型，也不是第二个角色扮演 Agent。它是位于 SillyTavern 正文与世界书之间的叙事记忆管理层。

```text
主模型：创作正文、扮演角色、推进剧情
镜渊：审核、提取、整理、压缩、恢复、发布记忆
```

## 2. 聊天隔离

每个聊天独立拥有：

- 状态快照；
- 内部事实；
- 小总结；
- 大总结；
- 历史失效和恢复检查点；
- 世界书映射；
- 唯一焦点对象；
- 任务状态。

所有异步任务都绑定 chatKey、消息身份和运行代次。任务经历聊天切换后，即使玩家再次回到原聊天，旧任务也不能重新获得提交资格。

## 3. 固定文本模型协议

模型不生成 JSON、数据库对象或稳定 ID。审核、状态和总结统一返回固定标签文本。

插件负责：

```text
解析标签
→ 校验字段
→ 匹配已有对象
→ 合并同轮重复项
→ 分配或继承稳定 ID
→ 保护未返回字段
→ 写入快照和事实账本
```

JSON 只用于插件内部设置、快照、任务、迁移和持久化。

## 4. 对象身份

长期记忆以对象为基本单位。默认对象包括时空、角色、物品、事件、区域、全局变化、基础设定和自定义对象。

同一聊天、同一表格、同一规范名称只能存在一个对象。不同经历写入同一对象的不同切面，而不是重复创建同名条目。

匹配顺序为：

```text
同表规范名称
→ 已登记别名和关键词
→ 历史稳定名称映射
→ 无匹配时创建新对象
```

模型始终不负责 ID。

## 5. 状态快照

每条成功处理的 AI 正文形成一份完整快照，表示截至当前消息结束时所有对象的有效状态。

本轮更新采用稀疏修订：

```text
复制上一轮完整快照
→ 更新模型明确返回的对象字段
→ 未返回对象保持不变
→ 未返回字段保持不变
→ 生成本轮新快照
```

对象定义、物品固有作用和历史层不会因为模型本轮省略而消失。

## 6. 对象目录与工作副本

状态模型不会接收完整数据库，而是接收两层旧信息：

- 全局对象短目录：表格、稳定名称和少量别名；
- 高相关对象工作副本：当前摘要、定义、事实、状态和必要关联。

目录只是身份地图，不是白名单。正文中新出现的对象仍可以返回，插件会在完整内部快照中复查后决定创建或合并。

## 7. 内部事实账本

快照回答“对象现在是什么状态”，内部事实回答“哪些事件导致了这些状态”。

内部事实记录事件线、已发生结果、未决事项、相关对象、来源消息、置信度和消费状态。它是总结和历史恢复的原料，不直接作为大量独立世界书条目发布。

## 8. 分层总结

### 小总结

小总结维护事件线的近期连续性。它结合上一版同事件小总结和新增事实，重写当前唯一有效版本。

保留：必要起因、关键转折、阶段结果、持续影响、近期未决事项。

删除：台词、动作流水账、气氛、普通反应、被结果覆盖的过程、失效和重复信息。

### 大总结

大总结不是缩写小总结，而是重新执行长期资格审查。

保留：长期结果、不可逆变化、长期关系/能力/归属变化、持续因果和跨阶段未决事项。

删除：事件经过、短期状态、临时地点、已解决问题和小总结的近义改写。

提示词决定信息判断和表达；插件代码决定事件分组、事实消费、版本继承、写入对象和事务提交。

## 9. 世界书发布

世界书是向主模型发布长期记忆的出口，不是镜渊内部数据库。

插件读取最新有效快照、近期经历和历史事实，按对象生成托管条目，并执行：

- 稳定条目身份复用；
- 名称唯一化；
- 当前聊天范围内的托管条目更新；
- constant、trigger、vector 模式设置；
- 总容量裁剪；
- 保存后回读和界面刷新。

焦点对象只影响 constant 属性。

## 10. 历史失效与断点恢复

历史编辑、删除或 Swipe 会使后续快照失效。镜渊记录最早失效位置，裁剪受影响的事实和总结关系，并暂停世界书发布。

重建按顺序执行，每条成功后保存检查点。中间失败时，下次从失败消息继续；已成功前缀不重复调用模型。

## 11. 任务队列与连接

审核、修正、状态、小总结和大总结进入 TaskQueue。相同任务键共享执行结果，待执行任务按优先级和 FIFO 排序。

模型请求按物理连接进入请求 lane。不同 Connection Profile 可以隔离调用，不会切换 SillyTavern 全局连接。

## 12. 核心原则

```text
模型负责理解
插件负责确定性管理

快照负责当前
事实负责依据
小总结负责近期连续性
大总结负责长期固化
世界书负责发布与召回

一个对象只有一个身份
每个聊天完全隔离
历史变化只重建受影响部分
```
