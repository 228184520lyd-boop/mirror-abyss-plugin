# 镜渊插件概念与业务逻辑

## 1. 定位

镜渊不是正文生成模型，也不是第二个角色扮演 Agent。它是位于 SillyTavern 正文与世界书之间的叙事记忆管理层。

```text
主模型：创作正文、扮演角色、推进剧情
镜渊：审核、提取、整理、压缩、恢复、发布记忆
```

## 2. 标准记忆分层

镜渊把记忆职责分开，避免同一份信息在每一层重复堆积：

```text
当前快照：对象此刻的有效状态
内部事实：导致状态变化的依据与来源
小总结：近期事件线的最小连续性
大总结：跨阶段仍成立的长期事实
关系图谱：明确对象、事件与关系的浏览视图
世界书：面向主模型的发布与召回出口
```

图谱不是额外数据库，世界书也不是内部事实账本。每一层只承担自己的信息资格。

## 3. 聊天隔离

每个聊天独立拥有状态快照、内部事实、小总结、大总结、历史检查点、世界书映射、唯一焦点对象和任务状态。

所有异步任务绑定 chatKey、消息身份和运行代次。任务经历聊天切换后，即使玩家再次回到原聊天，旧任务也不能重新获得提交资格。

## 4. 固定文本模型协议

模型不生成 JSON、数据库对象或稳定 ID。审核、状态和总结统一返回固定标签文本。

插件负责：

```text
解析标签
→ 校验字段
→ 匹配已有对象
→ 合并同轮重复项
→ 分配或继承稳定 ID
→ 保护未返回字段
→ 写入快照和事实账本
```

固定协议属于实现边界，只在诊断页只读展示；普通规则页只开放影响判断与表达的自然语言规则。

## 5. 对象准入与证据门槛

新对象必须具有后续使用价值，且能够稳定命名。仅出现姓名、服装、外貌、站位、情绪、短动作或一句台词不足以建档。

状态提取必须允许三种保守结果：

- 不建档；
- 不更新；
- 保留 reported / uncertain，不升级为 confirmed。

身份不清、同名无法确认、证据冲突或是否持续有效无法判断时，省略优先于猜测。只有正文明确推翻、结束或替换旧事实时，才更新既有状态。

## 6. 对象身份

长期记忆以对象为基本单位。默认对象包括时空、角色、物品、事件、区域、全局变化、基础设定和自定义对象。

同一聊天、同一表格、同一规范名称只能存在一个对象。不同经历写入同一对象的不同切面，而不是重复创建同名条目。

匹配顺序为：

```text
同表规范名称
→ 已登记别名和关键词
→ 历史稳定名称映射
→ 无匹配且通过准入规则时创建新对象
```

模型始终不负责 ID。

## 7. 状态快照与稀疏修订

每条成功处理的 AI 正文形成一份完整快照，表示截至当前消息结束时所有对象的有效状态。

```text
复制上一轮完整快照
→ 只更新明确返回且发生实质变化的字段
→ 未返回对象保持不变
→ 未返回字段保持不变
→ 生成本轮新快照
```

对象定义、物品固有作用和历史层不会因为模型本轮省略而消失。再次出现、重复描述和措辞变化不算更新。

## 8. 对象目录与工作副本

状态模型不会接收完整数据库，而是接收两层旧信息：

- 全局对象短目录：表格、稳定名称和少量别名；
- 高相关对象工作副本：当前摘要、定义、事实、状态和必要关联。

目录只是身份地图，不是白名单。正文中新出现且通过准入规则的对象仍可返回，插件会在完整内部快照中复查后决定创建或合并。

## 9. 内部事实账本

快照回答“对象现在是什么状态”，内部事实回答“哪些事件导致了这些状态”。

内部事实记录事件线、已发生结果、未决事项、相关对象、来源消息、置信度和消费状态。它是总结和历史恢复的原料，不直接作为大量独立世界书条目发布。

## 10. 分层总结

小总结维护事件线的近期连续性。它结合上一版同事件小总结和新增事实，重写当前唯一有效版本，只保留下一个阶段必须承接的最小事实集合。

大总结不是缩写小总结，而是重新执行长期资格审查，只固化跨场景、跨阶段仍会改变未来的已确认事实。

提示词决定信息判断和表达；插件代码决定事件分组、事实消费、版本继承、写入对象和事务提交。

## 11. 关系图谱

关系图谱从当前对象快照派生，并遵循显式关系优先：

```text
relationshipStates
→ relatedObjects / relatedEvents
→ 旧记录文本提及兼容回退
```

只要源对象已经有显式关联，图谱就不再从整段文本猜测其他连线。兼容回退边单独标记，便于人工清理旧数据。

图谱只读，不反向创建事实、关系或世界书条目。

## 12. 世界书发布

世界书是向主模型发布长期记忆的出口，不是镜渊内部数据库。

插件读取最新有效快照、近期经历和历史事实，按对象生成托管条目，并执行稳定条目身份复用、名称唯一化、当前聊天范围更新、召回模式设置、总容量裁剪、保存回读和界面刷新。

焦点对象只影响 constant 属性。

## 13. 历史失效与断点恢复

历史编辑、删除或 Swipe 会使后续快照失效。镜渊记录最早失效位置，裁剪受影响的事实和总结关系，并暂停世界书发布。

重建按顺序执行，每条成功后保存检查点。中间失败时，下次从失败消息继续；已成功前缀不重复调用模型。

## 14. 任务队列与连接

审核、修正、状态、小总结和大总结进入 TaskQueue。相同任务键共享执行结果，待执行任务按优先级和 FIFO 排序。

模型请求按物理连接进入请求 lane。不同 Connection Profile 可以隔离调用，不会切换 SillyTavern 全局连接。

## 15. UI 信息架构边界

普通页面只展示玩家任务：看状态、改规则、编辑对象、浏览关系、查看总结、发布记忆、设置自动化和排错。

底层固定标签、字段键、类型、稳定 ID、事实 ID、事件 ID 和迁移细节不进入普通流程。它们保留在代码或诊断页，避免简化 UI 反过来破坏确定性管理。

## 16. 核心原则

```text
模型负责理解
插件负责确定性管理

快照负责当前
事实负责依据
小总结负责近期连续性
大总结负责长期固化
图谱负责关系浏览
世界书负责发布与召回

证据不足可以不写
一个对象只有一个身份
每个聊天完全隔离
历史变化只重建受影响部分
```
