# rc.46 多事件身份与世界书迁移刷新修复报告

版本：`1.2.0-rc.46`  
流水线：`ma-pipeline-48`

## 修复一：多 eventIds 稳定身份继承

### 旧问题

`preserveStableObjectIds()` 只按单个主 `eventId` 建立旧对象索引。模型在下一轮改变 `eventIds` 顺序、临时 ID 和称呼时，即使新旧对象明确共享同一组事件，也可能无法继承旧稳定 ID，造成对象分裂和世界书重复。

### 当前规则

- 同时读取 `eventId` 与完整 `eventIds`，去重后建立事件索引；
- 只使用在新快照中唯一出现的事件 ID；
- 汇总全部未被占用的旧对象候选；
- 最终候选恰好一个时继承旧 ID 与锚点名称；
- 候选为零或多于一个时不猜测，继续使用标题、事实或新 ID 的既有安全链。

## 修复二：目标世界书切换后的旧书残留

### 旧问题

目标从世界书 A 改为 B 后，B 会发布新条目，但 A 中当前聊天的旧镜渊托管条目不会自动清理。

### 当前迁移顺序

1. 从 `ChatState.lastLorebookName` 取得上次实际发布书；
2. 解析当前目标书；
3. 对两本物理书按规范化名称固定顺序加锁；
4. 先向新目标书完整发布、保存、回读和刷新确认；
5. 再到旧书中只删除 `managed=true && chatKey===当前聊天` 的条目；
6. 人工条目、owner 未知条目、其他插件和其他聊天条目均不处理；
7. 旧书清理成功后才更新 `lastLorebookName`。

若旧书清理保存失败：

- 新书已成功入库的内容保留；
- 旧书落盘数据不丢失；
- 同步状态记为 failed；
- `lastLorebookName` 仍指向旧书；
- 下一次重试重新获取双书锁并完成清理。

## 修复三：世界书刷新与入库一致性

检查范围包括普通同步、强制同步、维护重发、旧书迁移、历史暂停和清空。

统一原则：

```text
生成/修改内存数据
→ saveWorldInfo
→ loadWorldInfo 回读
→ 再次对账并必要时补保存
→ reload/showWorldEditor
→ 再次 loadWorldInfo
→ 修正编辑器旧缓存回写
→ 必要时再次重绘
→ 最终回读确认
```

具体修复：

- 强制同步即使没有数据变化，也会实际刷新目标编辑器；
- 上一轮只在刷新阶段失败时，普通重试不会因“数据无变化”跳过刷新；
- 编辑器刷新若回写旧缓存产生重复、恢复已删除条目或取消暂停，插件会修正并再次重绘；
- 第二次重绘后仍持续回写时，最终入库会被修正，但任务明确失败并报告缓存无法稳定，不再伪报成功；
- 历史暂停和清空同时处理上次实际发布书与当前目标书，解决迁移中断期间双书残留或旧条目仍激活的问题；
- 所有删除和暂停继续严格按当前 `chatKey` 的镜渊托管元数据筛选。
